HPC-Diagnosis Agent v6 — 完整设计报告

从规则 Gate 升级为 Adversarial Sub-Agent 的完整设计稿，面向 SC'26 论文与工程实现
框架: LangGraph 核心升级: GATE Sub-Agent 论文 Claim: 零人工阈值 数据: formaltest + compositetest_backup

1. 项目定位

v6 的核心目标是把 Evidence Gating 从人工规则工程升级为双角色 LLM 对抗审查机制。主诊断者 THINK 负责提出和推进结论，GATE Sub-Agent 负责独立找漏洞，从而更好支撑“完全自主的 HPC 故障诊断 Agent 框架”这一论文定位。

核心 Claim
完全自主的 HPC 故障诊断 Agent 框架：零人工阈值、零标注数据、从原始指标到根因报告全程 LLM 驱动。

Hybrid Control

Triage 保留确定性压缩能力，Diagnosis 保留 LLM 推理弹性。

Adversarial Gating

GATE 不是被动判分器，而是主动寻找漏洞的审查子图。

Knowledge Accumulation

FPL 从空开始，通过 Reflect 在诊断过程中逐步成长。

Composite Diagnosis

显式维护多个候选根因，并区分独立故障与继发症状。

3. Triage 设计

Triage 继续保持 v5 的三步设计，不消耗 LLM token，但为后续主 Agent 和 GATE 提供结构化起点。

步骤内容产物
Step 1Z-score + 持久性过滤 + change pointTop-K 异常指标
Step 2按 onset 排序并聚合 subsystem scorecausal_order, leading_subsystem
Step 3与 jobinfo 对齐并构建上下文FocusContext
Triage 不是最终诊断器。它只负责压缩搜索空间，不能替代 THINK 或 GATE 的因果判断。

4. Diagnosis Agent

4.1 主 Agent

主 Agent 负责提出诊断假设并逐步收集证据。它的核心状态包括 hypotheses、evidence、react_trace、budget 与 gate_hint。主 Agent 可以使用三个工具：MetricQueryTool、KBRetrievalTool、DataAnalysisTool。

HYPOTHESIZE

输入 FocusContext 和 FPL 先验，生成 2–4 个候选假设。

THINK

根据当前证据空白与 gate_hint 决定下一步是补查还是 conclude。

ACT

执行工具调用并记录 budget 消耗。

OBSERVE

只做摘要压缩与 raw_stats 提取，不再打 strength 标签。

{
  "thought": "当前 h1 仍未排除 memory 替代解释，需要检查 swap 与时间先后关系",
  "action": {
    "type": "tool_call",
    "tool": "MetricQueryTool",
    "args": {},
    "reasoning": "验证 h1 的关键缺口"
  }
}

4.2 GATE Sub-Agent

GATE 是 v6 的关键升级：它是一个具备有限 ReAct 循环和受限工具集的 Adversarial Sub-Agent，而不是纯 Python 规则节点。

设计项v6 方案
角色证据审查员，目标是寻找漏洞而不是生成诊断
可见信息focus_context, hypotheses, evidence；不可见 THINK 的 thought
工具MetricQueryTool, KBRetrievalTool
预算独立 gate_budget，默认最多 3 次工具调用、2 轮循环
输出decision + reason + hint + diagnosis_type
快路径：GATE 可以零工具结束。如果第一轮逻辑审查就发现替代假设未排除，则直接 continue；如果没有关键漏洞，也可直接 pass。
system prompt 核心约束：
1. 你不是诊断者，你是审查者
2. 优先寻找逻辑缺口和反驳线索
3. 替代解释未排除时不得放行
4. 必须返回结构化 decision
5. 只有逻辑上无法裁决时才允许工具调用

4.3 THINK-GATE 交互协议

v6 采用方案 C：GATE 不直接把自身查到的结果并入主 evidence，而是以 gate_hint 形式告诉 THINK 下一步需要验证什么。

GATE 输出系统行为
pass直接进入 FINALIZE，gate_hint 置空
continue将 hint 写入 state，THINK 下一轮优先处理该提示
rehypothesize保留 refuted 假设，返回 HYPOTHESIZE 生成新假设
degrade进入 FINALIZE，输出 partial 或 inconclusive 报告
hint 必须是“下一步该查什么”的具体问题，不能是抽象否定。例如“请检查 memory 异常是否早于 cpu 异常出现”是好的 hint。

5. Knowledge Base

知识库仍维持两层结构：Metric KB 和 Fault Pattern Library。

层次内容v6 角色
Metric KB指标语义、normal_range、strength_thresholds、downstream_effects为 HYPOTHESIZE 和 GATE 提供领域参考
FPL结构化故障模式冷启动后逐步提高诊断效率
strength_thresholds 在 v6 中降级为参考文档，不再由 OBSERVE 确定性打标。GATE 会把它当作语义背景，而不是硬性裁决条件。

6. Evidence Gating

Evidence Gating 仍遵循三条原则，但实现从规则函数转为受控 LLM 审查。

证据独立性

防止同一现象被重复包装成多条支持证据。

反驳优先

一条强反驳可阻止通过，即使支持性证据很多。

复合故障甄别

区分独立根因与 derived symptoms。

GATE 的内部思考框架：
Stage A: 单假设是否已排除替代解释？
Stage B: 多假设是独立根因还是因果链？
Stage C: 应输出 pass / continue / rehypothesize / degrade？

v6 的关键不在于取消数字，而在于取消“外部工程师手写 pass 条件”。raw_stats、strength_thresholds 和 downstream_effects 仍然存在，但它们服务于 GATE 的审查思考，而非直接决定结果。

7. Reflect

Reflect 延续 v5 设计，只在存在可信 confirmed 根因时触发。它将成功诊断提炼为 FPL 规则，实现无监督知识积累。

diagnosis_typeReflect 策略
single_fault / composite_fault完整提炼
partial仅提炼 confirmed 假设，标记 incomplete
inconclusive跳过

8. 评估与消融

v6 的实验重点除了准确率和开销，还应特别回答 GATE Sub-Agent 是否真的值得其额外复杂度。

实验含义
Abl-A去掉 Triage，验证 Hybrid Control 价值
Abl-B去掉 GATE，验证独立审查必要性
Abl-C去掉 FPL，验证知识积累收益
Sup-C去掉 rehypothesize，验证纠错能力
Sup-DGATE 降级为单次 LLM call，验证主动调查能力
Sup-D 是 v6 特有的重要补充。它能直接回答：真正有价值的是“独立审查这个角色”，还是“GATE 拥有工具和循环的主动调查能力”。
