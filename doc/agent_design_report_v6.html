HPC-Diagnosis Agent v6 — 完整设计报告

基于 LLM Agent 的 HPC 系统异常诊断工具 · 版本 v6 · 面向 SC'26 的完整设计方案
框架: LangGraph 核心机制: Adversarial Evidence Gating 数据: 29 个 formaltest 实验 目标: SC'26

1. 项目背景

HPC 集群故障诊断的核心难点在于：指标维度高、异常传播具有时序链条、复合故障比例高，而且仅靠单次 LLM 调用容易出现“证据不足但仍然强行归因”的问题。当前方案在保持两阶段结构清晰的前提下，引入独立的审查者角色，使系统既能自主推理，也能主动质疑自己的结论。

系统定位
完全自主的 HPC 故障诊断 Agent 框架：零人工阈值、零标注数据、从原始指标到根因报告全程 LLM 驱动。

Hybrid Control

Triage 保留确定性压缩能力，Diagnosis 保留 LLM 推理弹性。

Adversarial Gating

GATE 不是被动判分器，而是主动寻找漏洞的审查子图。

Knowledge Accumulation

FPL 从空开始，通过 Reflect 在诊断过程中逐步成长。

Composite Diagnosis

显式维护多个候选根因，并区分独立故障与继发症状。

3. 设计展开

以下各节对应当前设计稿中的核心模块：Triage 负责压缩输入空间，Diagnosis Agent 负责提出和推进假设，GATE Sub-Agent 负责独立审查，Reflect 负责将成功经验写回 Fault Pattern Library。

3.1 Triage Stage

Triage 继续保持 v5 的三步设计，不消耗 LLM token，但为后续主 Agent 和 GATE 提供结构化起点。

步骤内容产物
Step 1Z-score + 持久性过滤 + change pointTop-K 异常指标
Step 2按 onset 排序并聚合 subsystem scorecausal_order, leading_subsystem
Step 3与 jobinfo 对齐并构建上下文FocusContext
Triage 不是最终诊断器。它只负责压缩搜索空间，不能替代 THINK 或 GATE 的因果判断。

3.2 Diagnosis Agent

主 Agent 负责提出诊断假设并逐步收集证据。它的核心状态包括 hypotheses、evidence、react_trace、budget 与 gate_hint。主 Agent 可以使用三个工具：MetricQueryTool、KBRetrievalTool、DataAnalysisTool。

状态字段类型作用
hypotheseslist[Hypothesis]显式维护多个候选根因及其状态
evidencelist[Evidence]主 Agent 收集的支持/反驳/中性证据
react_tracelist[ReActStep]THINK/ACT/OBSERVE 的可审计轨迹
gate_hintstr | nullGATE 回传的下一步验证问题
budgetdict主 Agent 的工具调用与 token 消耗

HYPOTHESIZE

输入 FocusContext 和 FPL 先验，生成 2–4 个候选假设。

THINK

根据当前证据空白与 gate_hint 决定下一步是补查还是 conclude。

ACT

执行工具调用并记录 budget 消耗。

OBSERVE

只做摘要压缩与 raw_stats 提取，不再打 strength 标签。

{
  "thought": "当前 h1 仍未排除 memory 替代解释，需要检查 swap 与时间先后关系",
  "action": {
    "type": "tool_call",
    "tool": "MetricQueryTool",

  }

3.3 GATE Sub-Agent

GATE 是一个具备有限 ReAct 循环和受限工具集的 Adversarial Sub-Agent，而不是纯规则节点。它不负责生成诊断，只负责从反方视角寻找漏洞、补查关键空白，并决定是否允许结论进入 FINALIZE。

设计项当前方案
角色证据审查员，目标是寻找漏洞而不是生成诊断
可见信息focus_context, hypotheses, evidence；不可见 THINK 的 thought
工具MetricQueryTool, KBRetrievalTool
预算独立 gate_budget，默认最多 3 次工具调用、2 轮循环
输出decision + reason + hint + diagnosis_type
快路径：GATE 可以零工具结束。如果第一轮逻辑审查就发现替代假设未排除，则直接 continue；如果没有关键漏洞，也可直接 pass。
system prompt 核心约束：
1. 你不是诊断者，你是审查者
2. 优先寻找逻辑缺口和反驳线索
3. 替代解释未排除时不得放行
4. 必须返回结构化 decision
5. 只有逻辑上无法裁决时才允许工具调用
字段约束设计意图
gate_evidence独立于主 evidence 列表避免角色混淆，保留审计边界
gate_hint仅在 continue 时非空用“问题”驱动 THINK，而不是直接给答案
gate_budget默认 3 次工具调用，2 轮循环避免 GATE 退化成第二个 diagnosis agent

3.4 Evidence Gating

系统采用方案 C：GATE 不直接把自身查到的结果并入主 evidence，而是以 gate_hint 形式告诉 THINK 下一步需要验证什么。这样既保留角色边界，也避免 THINK 重复摸索同一个漏洞。

GATE 输出系统行为
pass直接进入 FINALIZE，gate_hint 置空
continue将 hint 写入 state，THINK 下一轮优先处理该提示
rehypothesize保留 refuted 假设，返回 HYPOTHESIZE 生成新假设
degrade进入 FINALIZE，输出 partial 或 inconclusive 报告
原理含义设计意图
证据独立性同一现象不能被包装成多条独立支持证据防止模型“堆叠措辞”替代真实验证
反驳优先一条强反驳可阻止通过把审查重点放在漏洞而非支持分数累加
复合故障甄别区分独立根因与继发症状链避免把下游效应误判为第二根因
hint 必须是“下一步该查什么”的具体问题，不能是抽象否定。例如“请检查 memory 异常是否早于 cpu 异常出现”是好的 hint。

3.5 Reflect 与 FPL

Reflect 负责把成功诊断提炼为 FPL 规则，形成跨实验的知识积累闭环；知识库仍维持两层结构：Metric KB 和 Fault Pattern Library。

层次内容作用
Metric KB指标语义、normal_range、strength_thresholds、downstream_effects为 HYPOTHESIZE 和 GATE 提供领域参考
FPL结构化故障模式冷启动后逐步提高诊断效率

Reflect 触发前置条件

diagnosis_typeReflect 策略
single_fault / composite_fault完整触发，提炼所有 confirmed 假设
partial部分触发，只提炼 confirmed 假设，并标记 incomplete
inconclusive跳过，避免写入噪声规则

FPL 冷启动行为

字段有 FPL 时FPL 为空时
prior_confidence使用 pattern hit 的 confidence默认 0.30
required_verifications有结构化验证路线图由 THINK 自主探索
诊断效率轮次更少，验证路径更明确通常会多 2–4 轮迭代
Metric KB 中的 strength_thresholds 仅作为语义参考，OBSERVE 不再确定性打标签；最终证据质量由 GATE 在审查过程中综合判断。

4. 评估与消融

评估重点除了准确率与成本，还要回答 GATE Sub-Agent 是否值得其额外复杂度，因此消融实验必须覆盖“无 GATE”和“单次审查 GATE”两个对照。

证据独立性

防止同一现象被重复包装成多条支持证据。

反驳优先

一条强反驳可阻止通过，即使支持性证据很多。

复合故障甄别

区分独立根因与 derived symptoms。

GATE 的内部思考框架：
Stage A: 单假设是否已排除替代解释？
Stage B: 多假设是独立根因还是因果链？
Stage C: 应输出 pass / continue / rehypothesize / degrade？
实验作用验证对象
Abl-A去掉 Triage，验证 Hybrid ControlTriage 的 context 压缩与 causal_order 价值
Abl-B去掉 GATE，验证独立审查必要性Adversarial Evidence Gating
Abl-C去掉 FPL，验证知识积累收益Fault Pattern Library
Sup-C禁用 rehypothesize，验证纠错机制初始假设偏差下的恢复能力
Sup-DGATE 降级为单次 LLM call，验证主动调查贡献GATE 的工具调用与循环价值
Sup-D 是本版最关键的补充实验之一。它能直接回答：真正有效的是“独立审查者角色”，还是“GATE 拥有受限工具和循环后的主动调查能力”。

5. 实现路线

工程实现建议沿着“先主图、再最小 GATE、最后完整 GATE Sub-Agent”的顺序推进，使控制流、消融边界和代码演进保持一致。

阶段内容关键文件
Phase BTriage 三步实现agent/triage.py
Phase C-1Main Agent 最小可运行图agent/diagnosis.py
Phase C-2最小 GATE（单轮、零工具）agent/gate.py
Phase C-3完整 GATE Sub-Agent + hint 协议agent/gate.py, agent/state.py
Phase DReflect + 评估框架agent/reflect.py, eval/evaluate.py
推荐先让主 Agent 路由稳定，再引入最小 GATE 验证交互协议，最后扩展到带工具调用的完整 Sub-Agent。这样实现路径与论文叙事一致，也更容易做消融。
