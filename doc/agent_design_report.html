HPC-Diagnosis Agent v5  —  完整设计报告

基于 LLM Agent 的 HPC 系统异常诊断工具  ·  版本 v5  ·  目标投稿 SC'26
框架: LangGraph 1.1.3 LLM: deepseek/deepseek-chat via OpenRouter 数据: 29 个 formaltest 实验 Phase A: 已完成（v5 更新）

1. 项目背景

1.1 HPC 异常诊断的挑战

现代 HPC 集群运行 NPB（NAS Parallel Benchmarks）等大规模 MPI 作业，多节点协同计算时，单个节点上的 CPU 满载、内存高占用、网络丢包或磁盘 I/O 瓶颈，都会影响整个作业的完成。Prometheus 等监控系统会产生 150+ 时序指标，这些指标涵盖 CPU 使用率、内存分配、网络收发、磁盘 I/O 等多个子系统。

故障诊断面临以下核心困难：

  • 指标爆炸：150+ 指标中，真正与故障相关的通常只有 5−10 个，大量无关指标会干扰推理过程。
  • 复合故障：实验数据中约 75% 的场景为复合故障（如 CPU 满载 + 内存高负载同时注入），需要同时诊断多个根因。
  • 时序因果依赖：故障传播有链式结构——CPU 满载可能继发内存 page cache 压力；diskfill 导致写入阻塞可能引发 iowait 上升。区分"独立注入"与"继发症状"是正确诊断的关键。
  • 知识积累难：新的故障模式不断出现，纯静态规则系统无法及时更新，需要一个可持续积累的诊断知识库。

1.2 现有方案的局限性

方案类型代表系统主要局限
基于规则系统静态阈值规则规则需手工维护；无法处理未预见的故障组合；阈值对不同 workload 不通用
纯统计方法Z-score 异常检测只能发现"有异常"，不能解释"为什么"；误报率高
单次 LLM 调用本项目 Phase 2 基线输入全量 150+ 指标，大量 context 浪费；无中间验证步骤，容易"编故事式推理"；无法积累经验
STELLAR (SC'25)I/O 参数调优面向系统调优而非异常诊断；无 Evidence Gating，无多假设并行；知识库不区分规则来源/置信度

1.3 我们的核心切入点

四个核心设计贡献（每个均有对应消融实验支撑）：
  1. Hybrid Control：确定性 Triage（压缩 context，提取时序因果先验）+ LLM Agent（聚焦推理），两阶段逻辑独立，消融边界清晰。
  2. Evidence Gating：结论必须有量化证据支撑，防止 LLM 在证据不足时强行下结论。
  3. Fault Pattern Library（FPL）：初始为空，通过 Reflect 自动积累的结构化故障知识库。证明系统可以自行学习故障模式，无需人工预填规则。
  4. 多假设并行 + 继发症状甦别：AgentState 显式维护 N 个假设，cascade 检测区分“复合故障”与“继发症状链”。

3. 设计展开

以下各节对应 design/v5.md 的不同部分，并包含 design/kb_schema.md、design/pydantic_schema.md、design/evidence_gating.md 的细化内容。

3.1 知识库体系（两层结构）

知识库在系统启动前离线构建一次，不消耗推理时间，分为两层：

Metric KB

存储格式：YAML + ChromaDB 向量索引
覆盖 22 个关键指标，5 个子系统（CPU/Memory/Network/Disk/System）。每个条目包含语义描述、正常范围、strength_thresholds（weak/medium/strong × 值域 + 持续时间）、下游因果效应（downstream_effects，用于 cascade 判断）。

Fault Pattern Library (FPL)

存储格式：JSONL（每行一条规则）
初始为空，不预填任何人工规则，系统完全从零开始，通过 Reflect 节点在每次诊断后自动积累规则。每条规则包含症状签名 symptom_signature、可验证步骤 verification_steps、置信度和来源（learned）。

Triage 使用数据自适应基线——直接使用实验自身时序数据的前 N 个采样点计算统计基线，不依赖外部预计算的 Profile。这更符合真实部署场景，无需提前收集每种 workload 和节点配置下的正常态数据。
FPL 初始为空的设计理由：(1) 消融对比更干净——空 FPL vs 积累后的 FPL vs 无 FPL 三组；(2) 论文贡献更强——证明系统可自行学习故障模式；(3) strength 标签由 Metric KB 的 strength_thresholds 规则确定性打标，不由 LLM 主观判断，这是 Evidence Gating 可依赖的前提。

3.2 Triage Stage（Stage 1）

Triage 是纯确定性 Python 代码，不调用 LLM。职责：从 150+ 原始指标中压缩出 Top-K 异常指标，并输出带时序因果先验的 FocusContext。

处理流程（3 步）

  1. 异常指标筛选（Anomaly Scoring）：对每个指标做数据自适应 Z-score（使用实验自身时序数据的前 N 个采样点作为基线）+ 持久性过滤，可选增强变化点检测（PELT/BOCPD）标注 change_point。输出 Top-K 异常指标，每个附带 score、direction、t_onset
  2. 时序因果排序（Temporal Ordering）：按 t_onset 升序排列生成 causal_order；按 Metric KB subsystem 字段分组计算 subsystem_scores，确定 leading_subsystem
  3. FocusContext 构建：聚合所有信息；与 jobinfo.csv 做时间窗口交集关联作业和节点；计算 triage_confidence；生成完整 FocusContext 输入给 Stage 2

FocusContext 核心字段

字段类型说明
causal_orderlist[str]按 t_onset 升序排列的指标名列表，是 HYPOTHESIZE 的时序因果先验
leading_subsystemstr分诊初判的首发子系统，引导 HYPOTHESIZE 生成优先假设
top_metricslist[TopMetric]Top-K 异常指标，含 score/direction/t_onset/change_point/onset_rank
triage_confidencefloat [0,1]分诊可信度，影响 HYPOTHESIZE 生成的假设数量（低时生成更多候选）
subsystem_scoresdict各子系统加权异常总分，用于初步量化哪个系统问题最大
消融实验对应：Abl-A（无 Triage，全量 metrics 直接输入 HYPOTHESIZE），验证 Triage 的 context 压缩与时序因果先验效果。

3.3 Diagnosis Agent（Stage 2）

3.3.1 LangGraph 节点图

Agent 使用 LangGraph StateGraph 构建，状态类型为 AgentState（TypedDict）。节点间数据传递通过状态更新完成，所有中间推理步骤写入 react_trace 字段，支持完整审计。

节点类型职责输入输出（写入 State）
HYPOTHESIZELLM 节点基于 FocusContext + FPL 检索结果，生成 2–4 个候选假设FocusContext, FPL Top-K pattern hitshypotheses[], first_action
THINKLLM 节点（ReAct）分析当前证据状态，决定下一步工具调用或宣告结论hypotheses, evidence(摘要), 上次OBSERVE结果thought + action plan → ReActStep
ACT确定性节点按 THINK 输出执行工具调用，记录调用消耗action plantool raw output
OBSERVE确定性节点摘要压缩工具输出；按 strength_thresholds 打 strength 标签；更新 verification 状态tool raw output, Metric KBevidence[] append, hypothesis verification update
GATE纯 Python 节点三阶段规则判断：单假设评估 → cascade 甄别 → 全局决策hypotheses, evidence, budgetgate_result: decision/diagnosis_type
FINALIZELLM 节点按 DiagnosisReport schema 生成结构化最终报告confirmed hypotheses, evidence chainreport: DiagnosisReport

3.3.2 路由决策机制

THINK 节点通过 action.type 表达意图，GATE 节点负责规则验证，Budget Check 提供确定性兜底：

权责分离：THINK（LLM）回答"我是否准备好下结论了"——通过 action.type = "tool_call" | "conclude" 表达意图。GATE（规则）验证"你给出的结论证据够不够"——通过确定性规则判断。两者分离，防止 LLM 在证据不足时自信地下结论。

Budget 强制路由：每次进入 THINK 前，确定性检查 budget。若 budget 耗尽，无论 THINK 想做什么，强制路由到 GATE（带 remaining_budget=0 标记），GATE 据此输出 degrade 决策。

3.3.3 Pydantic Schema 结构（部分核心类）

class Hypothesis(BaseModel):
    id: str                         # h1, h2, h3...
    root_cause: str                 # 自然语言描述
    fault_type: str                 # 对应 FPL fault_type，用于 KB 检索
    subsystem: str                  # cpu | memory | network | disk
    prior_confidence: float         # FPL 命中规则的 confidence，无命中默认 0.30
    current_confidence: float       # OBSERVE 节点实时更新
    required_verifications: list[VerificationItem]  # 来自 FPL.verification_steps
    status: Literal["active", "confirmed", "refuted", "derived"]

class Evidence(BaseModel):
    id: str                         # e1, e2, e3...
    hypothesis_ids: list[str]       # 关联的假设 ID
    type: Literal["supporting", "refuting", "neutral"]
    source_tool: str                # MetricQueryTool | KBRetrievalTool | DataAnalysisTool
    query_summary: str              # 查询摘要（用于独立性去重）
    result_digest: str              # 1-2 句话的结果描述（进入 LLM context）
    raw_stats: dict                 # 统计量，不进入 context，用于 Gate 规则判断
    strength: Literal["strong", "medium", "weak"]   # 由 Metric KB 规则确定，非 LLM 主观

3.3.4 多假设并行（复合故障支持）

AgentState.hypotheses 是一个列表，可同时维护多个互不排斥的假设（如 h1=CPU故障, h2=内存故障）。THINK 节点在每轮决策时综合所有 active 假设的证据状态，决定优先验证哪个假设。

复合故障的 diagnosis_type 由 GATE 节点在 Stage C 根据 confirmed 假设数量确定：

  • single_fault：恰好 1 个 confirmed 假设，无 active
  • composite_fault：≥ 2 个独立 confirmed 假设，均通过 cascade 甄别
  • partial：≥ 1 个 confirmed，仍有 active（budget 耗尽时）
  • inconclusive：所有假设均 refuted 或 budget 耗尽且无 confirmed

3.3.5 工具摘要机制（防 token 爆炸）

OBSERVE 节点对工具原始输出执行摘要压缩：只保留统计量（p95/mean/max/duration 等）和 1-2 句文字摘要，写入 Evidence.result_digest；原始时序序列不进入 LLM context，仅用于离线调试。这保证了 10 次工具调用后 context 仍在 token 预算内。

3.4 Evidence Gating（核心贡献）

Evidence Gating 是防止"LLM 编故事"的核心机制。GATE 节点是纯 Python 规则，不消耗 LLM token。结论必须满足量化证据条件才能通过，否则继续收集证据或输出降级报告。

三项核心原理

原理内容设计意图
证据独立性要求结论必须有来自不同来源（不同工具、不同指标）的独立支持证据，单一信息源不足以确认假设防止 LLM 把同一现象反复包装成多条"证据"
反驳优先强反驳证据可直接否定假设，无论支持证据有多少体现科学推理"证伪优先"原则
复合故障甄别当多个假设通过时，基于时序同步性、因果链可达性和异常幅度区分独立故障与继发症状链防止将继发效应误判为复合故障

三阶段 Gate 决策框架

阶段内容关键逻辑
Stage A单假设评估基于独立证据数量、证据强度和 verification 完成状态，判定每个假设为 confirmed / refuted / active
Stage B继发症状甄别对 ≥2 confirmed 假设，检查时序同步性（近乎同时 → 独立注入）、因果链可达性（Metric KB downstream_effects）和异常幅度
Stage C全局决策输出 pass / continue / degrade 和最终 diagnosis_type（single_fault / composite_fault / partial / inconclusive）
具体阈值（独立证据来源数量、FPL 先验置信度要求等）作为可配置参数，不在设计文档中暴露，见 config/gate_config.yaml。

3.5 Reflect & 知识积累（Stage 3）

每次 FINALIZE 完成后触发 Reflect 节点（可同步或异步）。核心职责：将本次诊断经验提炼为 FPL 规则，实现无监督知识积累。

执行步骤

  1. 规则提炼：LLM 分析本次诊断的有效症状签名（leading_subsystem + anomaly metrics + temporal_pattern）和有效验证路径，按 FPL Schema 生成候选规则条目
  2. 去重写入：计算 symptom_signature 与已有规则的相似度，高相似度 → 版本号 +1（更新），否则 → 新增。新规则初始 status: "candidate"，confidence: 0.5
  3. 规则升级 / 衰减：候选规则被后续诊断成功匹配确认 ≥2 次 → 升为 "confirmed"；最近若干次表现不佳 → confidence 降低，标记 "deprecated"
Reflect 是无监督的——仅基于诊断过程中收集到的证据质量和一致性来提炼规则，无需知道"正确答案"。chaos.csv（注入记录）和 Ground Truth 比对放在独立的评估框架 eval/evaluate.py 中，不影响 Reflect 的规则积累逻辑。
防污染措施：LLM 只生成 symptom_signature、verification_steps、solutions 三个字段；confidence 由系统根据历史准确率计算（初始 0.5），不允许 LLM 直接设置。
