HPC-Diagnosis Agent v4  —  完整设计报告

基于 LLM Agent 的 HPC 系统异常诊断工具  ·  版本 v4  ·  目标投稿 SC'26
框架: LangGraph 1.1.3 LLM: deepseek/deepseek-chat via OpenRouter 数据: 29 个 formaltest 实验 Phase A: 已完成

1. 项目背景

1.1 HPC 异常诊断的挑战

现代 HPC 集群运行 NPB（NAS Parallel Benchmarks）等大规模 MPI 作业，多节点协同计算时，单个节点上的 CPU 满载、内存高占用、网络丢包或磁盘 I/O 瓶颈，都会影响整个作业的完成。Prometheus 等监控系统会产生 150+ 时序指标，这些指标涵盖 CPU 使用率、内存分配、网络收发、磁盘 I/O 等多个子系统。

故障诊断面临以下核心困难：

  • 指标爆炸：150+ 指标中，真正与故障相关的通常只有 5−10 个，大量无关指标会干扰推理过程。
  • 复合故障：实验数据中约 75% 的场景为复合故障（如 CPU 满载 + 内存高负载同时注入），需要同时诊断多个根因。
  • 时序因果依赖：故障传播有链式结构——CPU 满载可能继发内存 page cache 压力；diskfill 导致写入阻塞可能引发 iowait 上升。区分"独立注入"与"继发症状"是正确诊断的关键。
  • 知识积累难：新的故障模式不断出现，纯静态规则系统无法及时更新，需要一个可持续积累的诊断知识库。

1.2 现有方案的局限性

方案类型代表系统主要局限
基于规则系统静态阈值规则规则需手工维护；无法处理未预见的故障组合；阈值对不同 workload 不通用
纯统计方法Z-score 异常检测只能发现"有异常"，不能解释"为什么"；误报率高
单次 LLM 调用本项目 Phase 2 基线输入全量 150+ 指标，大量 context 浪费；无中间验证步骤，容易"编故事式推理"；无法积累经验
STELLAR (SC'25)I/O 参数调优面向系统调优而非异常诊断；无 Evidence Gating，无多假设并行；知识库不区分规则来源/置信度

1.3 我们的核心切入点

四个核心设计贡献（每个均有对应消融实验支撑）：
  1. Hybrid Control：确定性 Triage（压缩 context，提取时序因果先验）+ LLM Agent（聚焦推理），两阶段逻辑独立，消融边界清晰。
  2. Evidence Gating：结论必须有最低量化证据支撑（Rule A/B/C + 复合故障 Rule D），防止 LLM 在证据不足时强行下结论。
  3. Fault Pattern Library（FPL）：可积累、可写回的结构化故障知识库。每次诊断后通过 Reflect 节点自动提炼规则，以 JSONL 格式追加（带去重和置信度衰减机制）。
  4. 多假设并行 + 继发症状甄别：AgentState 显式维护 N 个假设，Stage B cascade 检测区分"复合故障"与"继发症状链"，解决 SC'26 复合故障场景的诊断难题。

3. 设计展开

以下各节对应 design/v4.md 的不同部分，并包含 design/kb_schema.md、design/pydantic_schema.md、design/evidence_gating.md 的细化内容。

3.1 知识库体系（Stage 0）

知识库在系统启动前离线构建一次，不消耗推理时间。由三层组成，各有不同的存储格式和使用场景：

Metric KB

存储格式：YAML + ChromaDB 向量索引
覆盖 22 个关键指标，5 个子系统（CPU/Memory/Network/Disk/System）。每个条目包含语义描述、正常范围、strength_thresholds（weak/medium/strong × 值域 + 持续时间）、下游因果效应（downstream_effects，用于 cascade 判断）。

Fault Pattern Library (FPL)

存储格式：JSONL（每行一条规则）
初始 8 条规则（fpl_001~008），覆盖 5 种单故障 + 2 种常见复合故障。每条包含症状签名 symptom_signature、可验证步骤 verification_steps、置信度、来源（manual/learned）和 composite_discrimination 字段。

Baseline Profile

存储格式：JSON（按实验条件分文件）
从 29 个 formaltest 实验的故障注入前 5 分钟提取正常态统计量（mean/std/p5/p95）。Fallback 优先级：精确匹配 → workload 类型匹配 → global 聚合 → 静态硬编码阈值，每级 fallback 对 triage_confidence 施加折减因子。

FPL 条目示例（fpl_007：CPU 满载 + 内存高负载复合故障）

{
  "pattern_id": "fpl_007",
  "fault_type": "composite:cpu_fullload+mem_load",
  "confidence": 0.82,
  "symptom_signature": {
    "leading_subsystem": "cpu",
    "required_metrics": ["cpu_usage_percent", "memory_usage_percent"],
    "causal_order_hint": ["cpu_usage_percent", "memory_usage_percent", "anon_memory_percent"]
  },
  "composite_discrimination": {
    "cascade_indicator":  "memory onset > 120s after CPU AND anon_memory increase < 15%  → single_fault:cpu_fullload",
    "composite_indicator":"anon_memory_percent rises > 20pp from baseline → composite_fault"
  },
  "verification_steps": [
    "confirm cpu_usage_percent p95 > 90% sustained > 3 min",
    "confirm memory_usage_percent > 80% sustained > 3 min",
    "confirm anon_memory_percent increased > 20pp from baseline (independent malloc)",
    "check onset time diff: |t_onset(cpu) - t_onset(memory)| < 120s supports co-injection"
  ]
}
设计亮点：strength 由 Metric KB 的 strength_thresholds 规则确定性打标，不由 LLM 主观判断。这是 Evidence Gating 可依赖的前提。

3.2 Triage Stage（Stage 1）

Triage 是纯确定性 Python 代码，不调用 LLM。职责：从 150+ 原始指标中压缩出 Top-K 异常指标，并输出带时序因果先验的 FocusContext。

处理流程

  1. 异常检测：对每个指标做 Z-score / 分位数偏离 / 静态阈值三重判定
  2. 持久性过滤：连续超阈值比例 > persistence_ratio 才保留（去除毛刺）
  3. 时序排序：记录每个异常指标首次超阈时间 t_onset，按时序升序排列为 causal_order（时序因果先验）
  4. 变化点检测：对 Top-K 异常指标做 PELT/BOCPD，标注 change_point 时刻
  5. 子系统分组：按 Metric KB subsystem 字段分组，计算 subsystem_scores，得出 leading_subsystem
  6. 作业关联：时间窗口与 jobinfo.csv 做交集，计算 overlap_ratio

FocusContext 核心字段

字段类型说明
causal_orderlist[str]按 t_onset 升序排列的指标名列表，是 HYPOTHESIZE 的时序因果先验（v4 新增贡献）
leading_subsystemstr分诊初判的首发子系统，引导 HYPOTHESIZE 生成优先假设
top_metricslist[TopMetric]Top-K 异常指标，含 score/direction/t_onset/change_point/onset_rank
triage_confidencefloat [0,1]分诊可信度，影响 HYPOTHESIZE 生成的假设数量（低时生成更多候选）
subsystem_scoresdict各子系统加权异常总分，用于初步量化哪个系统问题最大
消融实验对应：Ablation-A（无 Triage，全量 metrics 直接输入）和 Ablation-D（FocusContext 不含 causal_order）均可独立于 Stage 2 进行，分别验证"context 压缩效果"与"时序因果先验效果"。

3.3 Diagnosis Agent（Stage 2）

3.3.1 LangGraph 节点图

Agent 使用 LangGraph StateGraph 构建，状态类型为 AgentState（TypedDict）。节点间数据传递通过状态更新完成，所有中间推理步骤写入 react_trace 字段，支持完整审计。

节点类型职责输入输出（写入 State）
HYPOTHESIZELLM 节点基于 FocusContext + FPL 检索结果，生成 2–4 个候选假设FocusContext, FPL Top-3 pattern hitshypotheses[], first_action
THINKLLM 节点（ReAct）分析当前证据状态，决定下一步工具调用或宣告结论hypotheses, evidence(摘要), 上次OBSERVE结果thought + action plan → ReActStep
ACT确定性节点按 THINK 输出执行工具调用，记录调用消耗action plantool raw output
OBSERVE确定性节点摘要压缩工具输出；按 strength_thresholds 打 strength 标签；更新 verification 状态tool raw output, Metric KBevidence[] append, hypothesis verification update
GATE纯 Python 节点三阶段规则判断：单假设评估 → cascade 甄别 → 全局决策hypotheses, evidence, budgetgate_result: decision/diagnosis_type
FINALIZELLM 节点按 DiagnosisReport schema 生成结构化最终报告confirmed hypotheses, evidence chainreport: DiagnosisReport

3.3.2 Pydantic Schema 结构（部分核心类）

class Hypothesis(BaseModel):
    id: str                         # h1, h2, h3...
    root_cause: str                 # 自然语言描述
    fault_type: str                 # 对应 FPL fault_type，用于 KB 检索
    subsystem: str                  # cpu | memory | network | disk
    prior_confidence: float         # FPL 命中规则的 confidence，无命中默认 0.30
    current_confidence: float       # OBSERVE 节点实时更新
    required_verifications: list[VerificationItem]  # 来自 FPL.verification_steps
    status: Literal["active", "confirmed", "refuted", "derived"]

class Evidence(BaseModel):
    id: str                         # e1, e2, e3...
    hypothesis_ids: list[str]       # 关联的假设 ID
    type: Literal["supporting", "refuting", "neutral"]
    source_tool: str                # MetricQueryTool | KBRetrievalTool | DataAnalysisTool
    query_summary: str              # 查询摘要（用于独立性去重）
    result_digest: str              # 1-2 句话的结果描述（进入 LLM context）
    raw_stats: dict                 # 统计量，不进入 context，用于 Gate 规则判断
    strength: Literal["strong", "medium", "weak"]   # 由 Metric KB 规则确定，非 LLM 主观

3.3.3 多假设并行（复合故障支持）

AgentState.hypotheses 是一个列表，可同时维护多个互不排斥的假设（如 h1=CPU故障, h2=内存故障）。THINK 节点在每轮决策时综合所有 active 假设的证据状态，决定优先验证哪个假设。

复合故障的 diagnosis_type 由 GATE 节点在 Stage C 根据 confirmed 假设数量确定：

  • single_fault：恰好 1 个 confirmed 假设，无 active
  • composite_fault：≥ 2 个独立 confirmed 假设，均通过 cascade 甄别
  • partial：≥ 1 个 confirmed，仍有 active（budget 耗尽时）
  • inconclusive：所有假设均 refuted 或 budget 耗尽且无 confirmed

3.3.4 工具摘要机制（防 token 爆炸）

OBSERVE 节点对工具原始输出执行摘要压缩：只保留统计量（p95/mean/max/duration 等）和 1-2 句文字摘要，写入 Evidence.result_digest；原始时序序列不进入 LLM context，仅用于离线调试。这保证了 10 次工具调用后 context 仍在 token 预算内。

3.4 Evidence Gating（核心贡献）

Evidence Gating 是防止"LLM 编故事"的核心机制。GATE 节点是纯 Python 规则，不消耗 LLM token。结论必须满足量化证据条件才能通过，否则继续收集证据或输出降级报告。

三阶段 Gate 决策框架

阶段内容关键逻辑
Stage A单假设评估对每个 active 假设独立判断应升为 confirmed/refuted/active
Stage B继发症状甄别（Cascade Detection）对所有 ≥2 个 confirmed 假设两两检查是否存在因果从属关系，防止将继发效应误判为复合故障
Stage C全局决策输出 pass / continue / degrade 和最终 diagnosis_type

Stage A：单假设通过规则（Rule A / B / C / D）

规则通过条件适用场景
Rule A≥2 条来自不同工具+不同指标的支持证据，至少 1 条 strong标准场景，强调证据独立性
Rule B1 条 strong + 所有 required_verifications 完成（含 unverifiable）数据有缺失时的兜底
Rule CFPL 先验 > 0.80 + ≥1 条 verification 已核实 + 1 条 strong 证据高置信度历史规则辅助
Rule D（新增）当 ≥2 假设共享 >50% 支持证据时，置信度最高者胜出，另一者 refuted互斥假设（共享证据但解释矛盾）

Stage B：继发症状甄别算法（check_cascade）

对两个 confirmed 假设 (h_primary, h_secondary)，依次检查：

  1. 时序先后性：若 onset_lag < 60s（可配置）→ independent（几乎同时出现，说明独立注入）
  2. 因果链检查：查询 Metric KB downstream_effects，若 h_primary 的核心指标没有已知下游效应涉及 h_secondary 的子系统 → independent
  3. 异常幅度检查：若 h_secondary 的 p95 值 ≥ Metric KB 对应的 strong 阈值 → 继发效应不可能达到此幅度，判定 independent
  4. 特征量判断（CPU+MEM 特化）：anon_memory_percent 增幅 > 20pp → 独立 malloc 注入，判定 independent
  5. 以上均不通过 → cascade（h_secondary 降级为 derived）

可参数化阈值（均可在 config/gate_config.yaml 中覆盖，用于消融实验）

参数名默认值说明
min_independent_sources2Rule A 所需最少独立证据数
fpl_confidence_threshold0.80Rule C 生效的 FPL 先验阈值
cascade_lag_threshold_sec60Stage B：同时出现的时间窗口（秒）
anon_memory_independence_threshold20.0Stage B CPU+MEM：anon_memory 增幅独立性判断（百分点）
min_budget_per_hypothesis2继续处理每个 active 假设所需最低剩余工具调用次数
tool_calls_limit12单次诊断的工具调用上限（全局 budget）

3.5 Reflect & 知识积累（Stage 3）

每次 FINALIZE 完成后触发 Reflect 节点（可同步或异步）。核心职责：将本次诊断经验提炼为 FPL 规则，实现知识积累。

执行步骤

  1. Ground Truth 比对（评估模式）：将 root_causes 与 chaos.csv 中实际注入故障比对，输出 Hit@1 / Hit@K，写入 trace
  2. 规则提炼：LLM 分析本次诊断的"有效症状签名"与"有效验证路径"，按 FPL Schema 生成候选规则条目
  3. 去重写入：计算 symptom_signature 哈希，若与已有规则 cosine_similarity > 0.85 → 版本号 +1（更新），否则 → 新增（候选状态）
  4. 规则升级：候选规则被 ≥2 次成功诊断确认后升为正式规则（status: confirmed）
  5. 规则衰减：最近 5 次相关诊断准确率 < 40% → confidence 降低并标记 deprecated
防污染措施：LLM 只生成 symptom_signature、verification_steps、solutions 三个字段；confidence 由系统根据历史准确率计算（初始 0.5），不允许 LLM 直接设置。
