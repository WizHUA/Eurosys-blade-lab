1. 项目背景

1.1 HPC 异常诊断的挑战

现代 HPC 集群运行 NPB（NAS Parallel Benchmarks）等大规模 MPI 作业，多节点协同计算时，单个节点上的 CPU 满载、内存高占用、网络丢包或磁盘 I/O 瓶颈，都会影响整个作业的完成。Prometheus 等监控系统会产生 150+ 时序指标，这些指标涵盖 CPU 使用率、内存分配、网络收发、磁盘 I/O 等多个子系统。

故障诊断面临以下核心困难：

  • 指标爆炸：150+ 指标中，真正与故障相关的通常只有 5−10 个，大量无关指标会干扰推理过程。
  • 复合故障：实验数据中约 75% 的场景为复合故障（如 CPU 满载 + 内存高负载同时注入），需要同时诊断多个根因。
  • 时序因果依赖：故障传播有链式结构——CPU 满载可能继发内存 page cache 压力；diskfill 导致写入阻塞可能引发 iowait 上升。区分"独立注入"与"继发症状"是正确诊断的关键。
  • 知识积累难：新的故障模式不断出现，纯静态规则系统无法及时更新，需要一个可持续积累的诊断知识库。

1.2 现有方案的局限性

方案类型代表系统主要局限
基于规则系统静态阈值规则规则需手工维护；无法处理未预见的故障组合；阈值对不同 workload 不通用
纯统计方法Z-score 异常检测只能发现"有异常"，不能解释"为什么"；误报率高
单次 LLM 调用本项目 Phase 2 基线输入全量 150+ 指标，大量 context 浪费；无中间验证步骤，容易"编故事式推理"；无法积累经验
STELLAR (SC'25)I/O 参数调优面向系统调优而非异常诊断；无 Evidence Gating，无多假设并行；知识库不区分规则来源/置信度

1.3 我们的核心切入点

四个核心设计贡献（每个均有对应消融实验支撑）：
  1. Hybrid Control：确定性 Triage（压缩 context，提取时序因果先验）+ LLM Agent（聚焦推理），两阶段逻辑独立，消融边界清晰。
  2. Evidence Gating：结论必须有最低量化证据支撑（Rule A/B/C + 复合故障 Rule D），防止 LLM 在证据不足时强行下结论。
  3. Fault Pattern Library（FPL）：可积累、可写回的结构化故障知识库。每次诊断后通过 Reflect 节点自动提炼规则，以 JSONL 格式追加（带去重和置信度衰减机制）。
  4. 多假设并行 + 继发症状甄别：AgentState 显式维护 N 个假设，Stage B cascade 检测区分"复合故障"与"继发症状链"，解决 SC'26 复合故障场景的诊断难题。
