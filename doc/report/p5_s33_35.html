3.3 Diagnosis Agent（Stage 2）

3.3.1 LangGraph 节点图

Agent 使用 LangGraph StateGraph 构建，状态类型为 AgentState（TypedDict）。节点间数据传递通过状态更新完成，所有中间推理步骤写入 react_trace 字段，支持完整审计。

节点类型职责输入输出（写入 State）
HYPOTHESIZELLM 节点基于 FocusContext + FPL 检索结果，生成 2–4 个候选假设FocusContext, FPL Top-3 pattern hitshypotheses[], first_action
THINKLLM 节点（ReAct）分析当前证据状态，决定下一步工具调用或宣告结论hypotheses, evidence(摘要), 上次OBSERVE结果thought + action plan → ReActStep

GATE纯 Python 节点三阶段规则判断：单假设评估 → cascade 甄别 → 全局决策hypotheses, evidence, budgetgate_result: decision/diagnosis_type

3.3.2 Pydantic Schema 结构（部分核心类）

class Hypothesis(BaseModel):
    id: str                         # h1, h2, h3...
    root_cause: str                 # 自然语言描述
    fault_type: str                 # 对应 FPL fault_type，用于 KB 检索
    subsystem: str                  # cpu | memory | network | disk
    prior_confidence: float         # FPL 命中规则的 confidence，无命中默认 0.30
    current_confidence: float       # OBSERVE 节点实时更新
    required_verifications: list[VerificationItem]  # 来自 FPL.verification_steps
    status: Literal["active", "confirmed", "refuted", "derived"]

class Evidence(BaseModel):
    id: str                         # e1, e2, e3...
    hypothesis_ids: list[str]       # 关联的假设 ID
    type: Literal["supporting", "refuting", "neutral"]
    source_tool: str                # MetricQueryTool | KBRetrievalTool | DataAnalysisTool
    query_summary: str              # 查询摘要（用于独立性去重）
    result_digest: str              # 1-2 句话的结果描述（进入 LLM context）
    raw_stats: dict                 # 统计量，不进入 context，用于 Gate 规则判断
    strength: Literal["strong", "medium", "weak"]   # 由 Metric KB 规则确定，非 LLM 主观

3.3.3 多假设并行（复合故障支持）

AgentState.hypotheses 是一个列表，可同时维护多个互不排斥的假设（如 h1=CPU故障, h2=内存故障）。THINK 节点在每轮决策时综合所有 active 假设的证据状态，决定优先验证哪个假设。

复合故障的 diagnosis_type 由 GATE 节点在 Stage C 根据 confirmed 假设数量确定：

  • single_fault：恰好 1 个 confirmed 假设，无 active
  • composite_fault：≥ 2 个独立 confirmed 假设，均通过 cascade 甄别
  • partial：≥ 1 个 confirmed，仍有 active（budget 耗尽时）
  • inconclusive：所有假设均 refuted 或 budget 耗尽且无 confirmed

3.3.4 工具摘要机制（防 token 爆炸）

OBSERVE 节点对工具原始输出执行摘要压缩：只保留统计量（p95/mean/max/duration 等）和 1-2 句文字摘要，写入 Evidence.result_digest；原始时序序列不进入 LLM context，仅用于离线调试。这保证了 10 次工具调用后 context 仍在 token 预算内。

3.4 Evidence Gating（核心贡献）

Evidence Gating 是防止"LLM 编故事"的核心机制。GATE 节点是纯 Python 规则，不消耗 LLM token。结论必须满足量化证据条件才能通过，否则继续收集证据或输出降级报告。

三阶段 Gate 决策框架

阶段内容关键逻辑
Stage A单假设评估对每个 active 假设独立判断应升为 confirmed/refuted/active
Stage B继发症状甄别（Cascade Detection）对所有 ≥2 个 confirmed 假设两两检查是否存在因果从属关系，防止将继发效应误判为复合故障
Stage C全局决策输出 pass / continue / degrade 和最终 diagnosis_type

Stage A：单假设通过规则（Rule A / B / C / D）

规则通过条件适用场景
Rule A≥2 条来自不同工具+不同指标的支持证据，至少 1 条 strong标准场景，强调证据独立性
Rule B1 条 strong + 所有 required_verifications 完成（含 unverifiable）数据有缺失时的兜底
Rule CFPL 先验 > 0.80 + ≥1 条 verification 已核实 + 1 条 strong 证据高置信度历史规则辅助
Rule D（新增）当 ≥2 假设共享 >50% 支持证据时，置信度最高者胜出，另一者 refuted互斥假设（共享证据但解释矛盾）

Stage B：继发症状甄别算法（check_cascade）

对两个 confirmed 假设 (h_primary, h_secondary)，依次检查：

  1. 时序先后性：若 onset_lag < 60s（可配置）→ independent（几乎同时出现，说明独立注入）
  2. 因果链检查：查询 Metric KB downstream_effects，若 h_primary 的核心指标没有已知下游效应涉及 h_secondary 的子系统 → independent
  3. 异常幅度检查：若 h_secondary 的 p95 值 ≥ Metric KB 对应的 strong 阈值 → 继发效应不可能达到此幅度，判定 independent
  4. 特征量判断（CPU+MEM 特化）：anon_memory_percent 增幅 > 20pp → 独立 malloc 注入，判定 independent
  5. 以上均不通过 → cascade（h_secondary 降级为 derived）

可参数化阈值（均可在 config/gate_config.yaml 中覆盖，用于消融实验）

参数名默认值说明
min_independent_sources2Rule A 所需最少独立证据数
fpl_confidence_threshold0.80Rule C 生效的 FPL 先验阈值
cascade_lag_threshold_sec60Stage B：同时出现的时间窗口（秒）
anon_memory_independence_threshold20.0Stage B CPU+MEM：anon_memory 增幅独立性判断（百分点）
min_budget_per_hypothesis2继续处理每个 active 假设所需最低剩余工具调用次数
tool_calls_limit12单次诊断的工具调用上限（全局 budget）

3.5 Reflect & 知识积累（Stage 3）

每次 FINALIZE 完成后触发 Reflect 节点（可同步或异步）。核心职责：将本次诊断经验提炼为 FPL 规则，实现知识积累。

执行步骤

  1. Ground Truth 比对（评估模式）：将 root_causes 与 chaos.csv 中实际注入故障比对，输出 Hit@1 / Hit@K，写入 trace
  2. 规则提炼：LLM 分析本次诊断的"有效症状签名"与"有效验证路径"，按 FPL Schema 生成候选规则条目
  3. 去重写入：计算 symptom_signature 哈希，若与已有规则 cosine_similarity > 0.85 → 版本号 +1（更新），否则 → 新增（候选状态）
  4. 规则升级：候选规则被 ≥2 次成功诊断确认后升为正式规则（status: confirmed）
  5. 规则衰减：最近 5 次相关诊断准确率 < 40% → confidence 降低并标记 deprecated
防污染措施：LLM 只生成 symptom_signature、verification_steps、solutions 三个字段；confidence 由系统根据历史准确率计算（初始 0.5），不允许 LLM 直接设置。
