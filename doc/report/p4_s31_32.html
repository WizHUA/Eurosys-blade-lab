3. 设计展开

以下各节对应 design/v4.md 的不同部分，并包含 design/kb_schema.md、design/pydantic_schema.md、design/evidence_gating.md 的细化内容。

3.1 知识库体系（Stage 0）

知识库在系统启动前离线构建一次，不消耗推理时间。由三层组成，各有不同的存储格式和使用场景：

Metric KB

存储格式：YAML + ChromaDB 向量索引
覆盖 22 个关键指标，5 个子系统（CPU/Memory/Network/Disk/System）。每个条目包含语义描述、正常范围、strength_thresholds（weak/medium/strong × 值域 + 持续时间）、下游因果效应（downstream_effects，用于 cascade 判断）。

Fault Pattern Library (FPL)

存储格式：JSONL（每行一条规则）
初始 8 条规则（fpl_001~008），覆盖 5 种单故障 + 2 种常见复合故障。每条包含症状签名 symptom_signature、可验证步骤 verification_steps、置信度、来源（manual/learned）和 composite_discrimination 字段。

Baseline Profile

存储格式：JSON（按实验条件分文件）
从 29 个 formaltest 实验的故障注入前 5 分钟提取正常态统计量（mean/std/p5/p95）。Fallback 优先级：精确匹配 → workload 类型匹配 → global 聚合 → 静态硬编码阈值，每级 fallback 对 triage_confidence 施加折减因子。

FPL 条目示例（fpl_007：CPU 满载 + 内存高负载复合故障）

{
  "pattern_id": "fpl_007",
  "fault_type": "composite:cpu_fullload+mem_load",
  "confidence": 0.82,
  "symptom_signature": {
    "leading_subsystem": "cpu",
    "required_metrics": ["cpu_usage_percent", "memory_usage_percent"],
    "causal_order_hint": ["cpu_usage_percent", "memory_usage_percent", "anon_memory_percent"]
  },
  "composite_discrimination": {
    "cascade_indicator":  "memory onset > 120s after CPU AND anon_memory increase < 15%  → single_fault:cpu_fullload",
    "composite_indicator":"anon_memory_percent rises > 20pp from baseline → composite_fault"
  },
  "verification_steps": [
    "confirm cpu_usage_percent p95 > 90% sustained > 3 min",
    "confirm memory_usage_percent > 80% sustained > 3 min",
    "confirm anon_memory_percent increased > 20pp from baseline (independent malloc)",
    "check onset time diff: |t_onset(cpu) - t_onset(memory)| < 120s supports co-injection"
  ]
}
设计亮点：strength 由 Metric KB 的 strength_thresholds 规则确定性打标，不由 LLM 主观判断。这是 Evidence Gating 可依赖的前提。

3.2 Triage Stage（Stage 1）

Triage 是纯确定性 Python 代码，不调用 LLM。职责：从 150+ 原始指标中压缩出 Top-K 异常指标，并输出带时序因果先验的 FocusContext。

处理流程

  1. 异常检测：对每个指标做 Z-score / 分位数偏离 / 静态阈值三重判定
  2. 持久性过滤：连续超阈值比例 > persistence_ratio 才保留（去除毛刺）
  3. 时序排序：记录每个异常指标首次超阈时间 t_onset，按时序升序排列为 causal_order（时序因果先验）
  4. 变化点检测：对 Top-K 异常指标做 PELT/BOCPD，标注 change_point 时刻
  5. 子系统分组：按 Metric KB subsystem 字段分组，计算 subsystem_scores，得出 leading_subsystem
  6. 作业关联：时间窗口与 jobinfo.csv 做交集，计算 overlap_ratio

FocusContext 核心字段

字段类型说明
causal_orderlist[str]按 t_onset 升序排列的指标名列表，是 HYPOTHESIZE 的时序因果先验（v4 新增贡献）
leading_subsystemstr分诊初判的首发子系统，引导 HYPOTHESIZE 生成优先假设
top_metricslist[TopMetric]Top-K 异常指标，含 score/direction/t_onset/change_point/onset_rank
triage_confidencefloat [0,1]分诊可信度，影响 HYPOTHESIZE 生成的假设数量（低时生成更多候选）
subsystem_scoresdict各子系统加权异常总分，用于初步量化哪个系统问题最大
消融实验对应：Ablation-A（无 Triage，全量 metrics 直接输入）和 Ablation-D（FocusContext 不含 causal_order）均可独立于 Stage 2 进行，分别验证"context 压缩效果"与"时序因果先验效果"。
