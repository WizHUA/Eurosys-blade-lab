# HPC-Diagnosis Agent Design (v6)

本文是 HPC-Diagnosis Agent 的完整系统设计规格，面向 SC'26 论文写作。本文档给出当前方案的完整定义，包括系统架构、状态设计、工具契约、Evidence Gating、Reflect 与评估方案。版本变更历史见 `design/changelog.md`。

---

## Part I：设计原则

1. **两阶段清晰分离**：Triage 与 Agent 层职责独立，便于消融与复现
2. **双 Agent 对抗审查**：Diagnosis Agent 负责诊断推进，Audit Agent 负责独立质疑，两者由 Orchestrator 协调且架构级隔离，防止自我背书
3. **诊断结论零硬编码阈值**：不以人工规则决定结论是否成立。Triage 的统计检测超参数属于信号抽取层，诊断裁决层由 Audit Agent LLM 自主判断
4. **多假设并行跟踪**：显式维护多个候选根因及其各自证据链
5. **工具输出摘要化**：原始时序不直接进入长上下文，避免 token 爆炸
6. **规则库有 Schema**：FPL 只能按固定 JSON Schema 回写，防止知识污染
7. **可审计执行轨迹**：Diagnosis Trace 与 Audit Trace 分离记录，便于分析失败模式
8. **术语约定**：`causal_order` 是按异常 onset 时间排序的序列，作为因果推断的先验提示，不等同于严格因果关系（temporal ordering as causal prior）

## Part II：系统架构

### 2.1 输入 / 输出

| 方向 | 内容 | 格式 |
|------|------|------|
| 输入 | `metrics.csv` — 时序指标矩阵（~150 列） | CSV，时间戳索引 |
| 输入 | `jobinfo.csv` — Slurm 作业信息 | CSV |
| 输出 | `DiagnosisReport` — 根因列表、证据链、解决建议 | JSON |

> `chaos.csv` 仅用于评估框架（Part VIII），不参与诊断流程。

### 2.3 与 STELLAR (SC'25) 的对比定位

| 维度 | STELLAR (SC'25) | 本系统 |
|------|-----------------|--------|
| 应用目标 | I/O 参数自动调优 | 故障根因诊断 |
| 推理控制 | LLM 自由生成 + 反思总结 | **确定性 Triage** + 双 Agent 对抗架构 |
| 证据验证 | 无独立审查者 | **独立 Audit Agent + Adversarial Evidence Gating** |
| 多故障支持 | 单目标优化 | **多假设并行**，支持复合故障 |
| 知识库 | RAG 文档检索 + Rule Set 积累 | **结构化 FPL**（含状态、版本、来源追踪） |

### 2.4 核心贡献点

1. **Hybrid Control**：确定性 Triage 压缩 context、提供 causal_order，主 Diagnosis Agent 负责聚焦推理
2. **Adversarial Evidence Gating**：Diagnosis Agent 与 Audit Agent 架构级分离、独立 State 信息隔离，Audit Agent 以对抗审查者身份主动寻找漏洞，实现诊断结论零硬编码阈值的证据门控
3. **Fault Pattern Library**：从空开始，通过 Reflect 自动积累的结构化故障知识库
4. **多假设并行 + 继发症状甄别**：显式维护 N 个候选根因，并区分复合故障与下游症状链

### 2.5 论文 Claim

> 完全自主的 HPC 故障诊断 Agent 框架：诊断结论零硬编码阈值、零标注数据、诊断裁决由 LLM Agent 驱动。

这里的"诊断裁决由 LLM Agent 驱动"并不否认 Triage 的确定性实现，而是强调最终诊断裁决不依赖人工编写的阈值规则，不需要研究者手工定义"多少条 strong 证据才算通过"。

## Part III：Stage 1 — Triage

Triage 是纯确定性 Python 代码，不调用 LLM，不消耗 token。职责：从 150+ 原始指标中筛选 Top-K 异常指标，输出带时序因果先验的 `FocusContext`。

### 3.1 处理流程（3 步）

#### Step 1：异常指标筛选（Anomaly Scoring）

对每个指标列执行：

- **数据自适应基线**：使用实验自身时序数据计算统计基线（前 N 个采样点的均值/标准差，或整体滑动窗口统计量），不依赖外部预计算的 Baseline Profile
- **Z-score 异常评分**：相对于数据自适应基线计算 Z-score
- **持久性过滤**：连续超阈值比例未达到持久性要求的指标剔除，去除毛刺
- **变化点检测**：对候选异常指标做 PELT 或 BOCPD，标注 `change_point`，精确定位 `t_onset`

输出：Top-K 异常指标，每个指标附带 `score`、`direction`、`t_onset`。

#### Step 2：时序因果排序（Temporal Ordering）

- 按 `t_onset` 升序排列异常指标，生成 `causal_order`
- 按 Metric KB 的 `subsystem` 字段分组，计算 `subsystem_scores`
- 确定 `leading_subsystem`

#### Step 3：FocusContext 构建

- 将 Top-K 指标、causal_order、subsystem_scores 聚合
- 与 `jobinfo.csv` 做时间窗口交集，关联受影响的作业和节点
- 计算 `triage_confidence`

### 3.2 FocusContext Schema

```json
{
  "run_id": "string",
  "anomaly_window": {"start": "ISO8601", "end": "ISO8601"},
  "top_metrics": [
    {
      "metric": "string",
      "subsystem": "string",
      "direction": "+|-",
      "score": 0.0,
      "t_onset": "ISO8601",
      "onset_rank": 1,
      "change_point": "ISO8601|null"
    }
  ],
  "causal_order": ["metric_A", "metric_B", "metric_C"],
  "subsystem_scores": {"cpu": 0.0, "memory": 0.0, "network": 0.0, "disk": 0.0},
  "leading_subsystem": "string",
  "nodes": [{"node": "string", "severity": "high|medium|low"}],
  "jobs": [{"job_id": "string", "overlap_ratio": 0.0, "node_set": []}],
  "triage_confidence": 0.0
}
```

关键字段：

- `causal_order`：按 `t_onset` 排序的指标列表，是时序因果先验
- `leading_subsystem`：分诊初判的首发子系统
- `triage_confidence`：分诊可信度，低时 HYPOTHESIZE 会生成更多候选假设

## Part IV：Stage 2 — Orchestrator { Diagnosis Agent ↔ Audit Agent }

### 4.1 三层状态设计

Stage 2 包含三个独立的 State，分别归属 Orchestrator、Diagnosis Agent 和 Audit Agent。状态之间通过 `ConclusionProposal` 和 `AuditDecision` 两个结构化消息传递信息，实现架构级信息隔离。

#### OrchestratorState

```python
class OrchestratorState(TypedDict):
    run_id: str
    inputs: dict                        # {metrics_path, jobinfo_path}
    focus_context: FocusContext         # Stage 1 产物，传递给两个 Agent
    current_proposal: ConclusionProposal | None  # Diagnosis Agent 的最新结论提案
    audit_decision: AuditDecision | None          # Audit Agent 的最新裁决
    rehyp_count: int                    # 重新假设次数（Orchestrator 全局计数）
    round_count: int                    # Diagnosis-Audit 交互轮次
    report: DiagnosisReport | None      # FINALIZE 产物
```

#### DiagnosisState

```python
class DiagnosisState(TypedDict):
    run_id: str
    focus_context: FocusContext         # Orchestrator 传入的只读副本
    hypotheses: list[Hypothesis]        # 多假设并行
    evidence: list[Evidence]            # Diagnosis Agent 收集的证据池
    react_trace: list[ReActStep]        # THINK/ACT/OBSERVE 轨迹
    gate_hint: str | None               # Orchestrator 注入的 Audit Agent 反馈
    budget: dict                        # {tool_calls_used, tool_calls_limit, tokens_in, tokens_out}
    rehyp_count: int                    # Orchestrator 传入（只读）
```

#### AuditState

```python
class AuditState(TypedDict):
    run_id: str
    focus_context: FocusContext         # Orchestrator 传入的只读副本
    proposal: ConclusionProposal       # Orchestrator 提交的结论提案（唯一可见的诊断信息）
    audit_evidence: list[AuditEvidence] # Audit Agent 独立收集的证据池
    audit_trace: list[AuditStep]        # GATE_THINK/ACT/OBSERVE 轨迹
    audit_budget: dict                  # {tool_calls_used, tool_calls_limit, max_rounds}
    previous_hint: str | None           # 最近一次 hint（如果本轮是重审）
```

#### ConclusionProposal — 信息隔离边界

```python
class ConclusionProposal(BaseModel):
    """Diagnosis Agent → Orchestrator → Audit Agent 的唯一信息通道"""
    hypotheses: list[Hypothesis]        # 假设列表（含 status、confidence）
    evidence: list[Evidence]            # Diagnosis Agent 收集的证据池
    proposed_diagnosis_type: str        # single_fault | composite_fault | partial
    proposed_root_causes: list[dict]    # [{cause, fault_type, confidence, evidence_ids}]
```

ConclusionProposal 不包含：

- `react_trace`（Diagnosis Agent 的推理轨迹）
- THINK 节点的 `thought` 字段
- Diagnosis Agent 的 system prompt 或内部模板
- `budget` 详情

这确保了 Audit Agent 只能看到"结论是什么 + 支撑证据是什么"，而看不到"是怎么推导出来的"，从而实现真正的信息隔离。

#### AuditDecision

```python
class AuditDecision(BaseModel):
    """Audit Agent → Orchestrator 的裁决输出"""
    decision: Literal["pass", "continue", "rehypothesize", "degrade"]
    reason: str
    hint: str | None                    # 仅 continue 时非空
    diagnosis_type: str | None          # single_fault | composite_fault | partial | inconclusive | null
```

设计要点：

- **Diagnosis Agent 的 `evidence` 与 Audit Agent 的 `audit_evidence` 物理隔离**：分属不同 State 对象，不存在跨 Agent 直接访问
- **`gate_hint`** 是 Orchestrator 从 AuditDecision 中提取后注入 DiagnosisState 的桥接字段，承载"下一步该查什么"
- **Budget 独立**：`budget`（诊断）与 `audit_budget`（审计）分离，支持独立统计与消融
- **Orchestrator 不持有证据**：OrchestratorState 不保存证据池，仅传递结构化消息

### 4.2 Hypothesis / Evidence Schema

```python
class Hypothesis(BaseModel):
    id: str
    root_cause: str
    fault_type: str
    subsystem: str
    prior_confidence: float
    current_confidence: float
    required_verifications: list[VerificationItem]
    status: Literal["active", "confirmed", "refuted", "derived"]

class Evidence(BaseModel):
    id: str
    hypothesis_ids: list[str]
    type: Literal["supporting", "refuting", "neutral", "mixed"]
    source_tool: str
    query_summary: str
    result_digest: str
    raw_stats: dict

class AuditEvidence(BaseModel):
    id: str
    source_tool: str
    query_summary: str
    result_digest: str
    raw_stats: dict
    target_hypothesis_ids: list[str]
    purpose: str
```

关键设计：

- `Evidence.strength` 字段被移除
- OBSERVE 不再给证据打 `strong / medium / weak` 标签
- `raw_stats` 保留，供 Audit Agent LLM 根据上下文自主判断证据质量

#### current_confidence 更新机制

v6 删除 `Evidence.strength` 后，confidence 更新不再依赖确定性计分公式。改由如下机制：

- **THINK 每轮更新**：THINK 在每轮 thought 输出中报告各假设的 confidence 估计值，写入 `Hypothesis.current_confidence`
- **Audit Agent 校准**：若 Audit Agent 发现 confidence 与证据不符（如明显高估），在 `hint` 中指出
- **约束**：confidence 不允许在无新证据的情况下上升；FINALIZE 使用最终一轮 THINK 的 confidence

### 4.3 LangGraph 节点与路由

本系统由三个独立的 LangGraph StateGraph 组成：`diagnosis_graph`、`audit_graph` 和 `orchestrator_graph`。

#### Diagnosis Agent 节点一览（diagnosis_graph）

| 节点 | 类型 | 职责 |
|------|------|------|
| HYPOTHESIZE | LLM | 基于 FocusContext + FPL Top-K 命中生成 2–4 个候选假设 |
| THINK | LLM (ReAct) | 结合证据状态与 `gate_hint` 决定下一步调查或尝试下结论 |
| ACT | 确定性 | 执行工具调用，记录资源消耗 |
| OBSERVE | 确定性 | 压缩工具输出、生成 `result_digest`、提取 `raw_stats`、更新 verification 状态 |

#### Audit Agent 节点一览（audit_graph）

| 节点 | 类型 | 职责 |
|------|------|------|
| GATE_THINK | LLM | 独立审查 ConclusionProposal，寻找漏洞，必要时发起工具调用 |
| GATE_ACT | 确定性 | 执行受限工具调用 |
| GATE_OBSERVE | 确定性 | 生成 `audit_evidence`，回到 GATE_THINK |

#### Orchestrator 节点一览（orchestrator_graph）

| 节点 | 类型 | 职责 |
|------|------|------|
| INVOKE_DIAGNOSIS | 确定性 | 调用 diagnosis_graph.invoke()，获取 ConclusionProposal |
| SUBMIT_TO_AUDIT | 确定性 | 将 ConclusionProposal 封装后提交给 audit_graph.invoke() |
| ROUTE_DECISION | 确定性 | 根据 AuditDecision 路由到 FINALIZE / 继续诊断 / 重开假设 |
| FINALIZE | LLM | 生成最终 DiagnosisReport |

#### 路由规则

**Diagnosis Agent 内部路由（diagnosis_graph）：**

| 来源 | 条件 | 目标 |
|------|------|------|
| HYPOTHESIZE | 完成 | → THINK |
| THINK | `action.type = "tool_call"` | → ACT |
| THINK | `action.type = "conclude"` | → 返回 ConclusionProposal（图终止） |
| ACT | 完成 | → OBSERVE |
| OBSERVE | 完成 | → THINK |
| ACT/OBSERVE | budget exhausted | → THINK（强制 conclude） |

**Audit Agent 内部路由（audit_graph）：**

| 来源 | 条件 | 目标 |
|------|------|------|
| GATE_THINK | 逻辑审查已得出裁决 | → 返回 AuditDecision（图终止） |
| GATE_THINK | 需要补查 | → GATE_ACT |
| GATE_ACT | 完成 | → GATE_OBSERVE |
| GATE_OBSERVE | 完成 | → GATE_THINK |

**Orchestrator 路由（orchestrator_graph）：**

| 来源 | 条件 | 目标 |
|------|------|------|
| INVOKE_DIAGNOSIS | Diagnosis Agent 返回 ConclusionProposal | → SUBMIT_TO_AUDIT（强制） |
| SUBMIT_TO_AUDIT | Audit Agent 返回 AuditDecision | → ROUTE_DECISION |
| ROUTE_DECISION | `decision = "pass"` | → FINALIZE |
| ROUTE_DECISION | `decision = "continue"` | → 注入 hint 到 DiagnosisState → INVOKE_DIAGNOSIS |
| ROUTE_DECISION | `decision = "rehypothesize"` | → 重置部分 DiagnosisState → INVOKE_DIAGNOSIS |
| ROUTE_DECISION | `decision = "degrade"` | → FINALIZE |

**Budget 保护**：当 `budget.tool_calls_used >= budget.tool_calls_limit` 时，THINK 的 system prompt 中注入强制指令——此轮必须输出 `action.type = "conclude"`。若 THINK 仍违规尝试 tool_call，ACT 节点拒绝执行并将 THINK 的输出改写为 conclude。

**权责分离**：Diagnosis Agent 回答"我是否准备好下结论"；Audit Agent 回答"这个结论是否经得起独立审查"。两者是独立的 LangGraph StateGraph，信息隔离由架构保证而非代码约定。Orchestrator 是确定性协调者，不参与诊断推理。



#### THINK 节点输出格式

```json
{
  "thought": "e1 和 e2 说明 CPU 异常早于 memory 异常，h2 正在被削弱...",
  "action": {
    "type": "tool_call | conclude",
    "tool": "MetricQueryTool",
    "args": {},
    "reasoning": "需要验证 h1 的 verification[1]"
  }
}
```

### 4.4 HYPOTHESIZE 重新进入行为

当 Audit Agent 触发 `rehypothesize` 时，HYPOTHESIZE 节点被第二次调用。此时 state 中已包含：

- refuted 假设列表及其驳回理由
- 第一轮 Diagnosis Agent 已收集的 evidence
- FocusContext 原始结构化摘要

HYPOTHESIZE 的 prompt 在 `rehyp_count > 0` 时包含额外指令：

> 以下候选假设已经被前一轮审查否定，请不要重复这些 fault_type。请基于未被解释的异常指标和现有证据空白生成新的候选假设。

设计约束：

- 新假设 ID 续编（如 h5, h6）
- 旧假设保留在 `hypotheses` 列表中，`status = refuted`
- 新假设不得与已 refuted 假设的 `fault_type` 重复

### 4.5 复合故障支持

`AgentState.hypotheses` 可同时维护多个互不排斥的假设。THINK 在每轮对所有 active 假设进行比较，优先验证：

1. 与 `leading_subsystem` 更一致的假设
2. 所需 verification gap 更小的假设
3. 能解释更多 top_metrics 的假设

`diagnosis_type` 的定义：

- `single_fault`：恰好 1 个 confirmed，无 active
- `composite_fault`：≥2 个独立 confirmed，均经 Audit Agent 甄别后保留
- `partial`：≥1 个 confirmed，但 budget 耗尽，仍有 active
- `inconclusive`：无 confirmed，或全部 refuted 且无法继续

以上优先级以 prompt 约束方式传递给 THINK LLM，不由系统代码强制执行。在消融实验中，可通过比较有无优先级提示来评估其效果。

### 4.6 工具契约

#### MetricQueryTool

```text
输入：metrics, time_window, aggregation (mean|p95|max|duration_above_threshold)
输出：results [{metric, value, unit}], missing [{metric, reason}]
注：LLM 接收的是聚合摘要，不是原始序列
```

参数规格：
- `time_window`：格式为 `{start: ISO8601, end: ISO8601}`
- `duration_above_threshold`：聚合类型选择此项时，需额外提供 `threshold_value` 参数

#### KBRetrievalTool

```text
输入 mode A — metric_lookup：按指标名查询 Metric KB 条目
输入 mode B — pattern_match：按症状签名查询 FPL（子系统 + 异常指标 + causal_order）
输出：metric_entries[] 或 pattern_hits[]
```

参数规格：
- `pattern_match` 输入格式：`{subsystem: str, anomaly_metrics: list[str], causal_order: list[str]}`

#### DataAnalysisTool

```text
输入：analysis_type (correlation | changepoint | group_compare | lag_analysis)
输出：findings [{statistic_name, value, interpretation}], summary
注：禁止返回原始序列，只返回结构化统计量和文字摘要
```

参数规格：
- `lag_analysis` 需要：`{metric_a: str, metric_b: str, time_window: {start: ISO8601, end: ISO8601}}`

### 4.7 Audit Agent 详细设计

Audit Agent 不是一个简单的“判分节点”，而是一个受控的对抗审查子图。它的设计目标不是自行生成诊断，而是在 Diagnosis Agent 已提出结论时主动寻找反例、逻辑空白或未排除的替代解释。

#### 角色定义

- **身份**：Evidence Reviewer / Adversarial Auditor
- **目标**：尽可能推翻 THINK 当前结论；若找不到漏洞，才放行
- **工作原则**：先找逻辑缺口，再决定是否需要额外工具调用

#### Audit Agent 可见信息

Audit Agent 输入上下文只包含：

- `focus_context`
- `hypotheses`
- `evidence`
- `gate_hint` 的历史值（如果实现上保留最近一次）
- Metric KB 与 FPL 的必要检索结果

Audit Agent 不可见：

- `react_trace.thought`
- THINK 的完整 reasoning chain
- Diagnosis Agent 的 prompt 内部模板

这实现了信息隔离，避免 Audit Agent 顺着 THINK 的推理链走向“默认认同”。

#### Audit Agent System Prompt 结构

Audit Agent system prompt 建议由四段组成：

1. **角色声明**：你是证据审查员，不负责给出诊断，只负责寻找漏洞
2. **审查原则**：
   - 单一现象不能被重复包装成多条支持证据
   - 替代假设未排除时不得放行
   - 强反驳证据优先于支持性描述
   - 多根因情形必须区分独立故障与下游症状
3. **工具使用约束**：只有在逻辑检查不能直接得出裁决时才允许调用工具；最多 3 次
4. **输出要求**：必须返回 `decision`、`reason`、`hint`

#### Audit Agent 内部 ReAct 循环

```text
输入：结论候选 + evidence[] + hypotheses[]
    │
    ▼
GATE_THINK
  - 先做无工具的逻辑审查
  - 若已发现明确漏洞：continue / rehypothesize
  - 若已确认没有漏洞：pass
  - 若仍有关键不确定点：发起 gate_tool_call
    │
    ▼
GATE_ACT
  - 执行受限工具调用
    │
    ▼
GATE_OBSERVE
  - 生成 audit_evidence
  - 回到 GATE_THINK
```

#### 工具约束

Audit Agent 只允许使用：

- `MetricQueryTool`
- `KBRetrievalTool`

默认不允许使用 `DataAnalysisTool`，原因是：

- DataAnalysisTool 容易将 Audit Agent 变成第二个诊断者，而不是审查者
- Audit Agent 的职责是验证漏洞，不是重新跑一套完整分析

#### Budget 配置

```yaml
audit_budget:
  tool_calls_limit: 3
  max_rounds: 2
```

解释：

- 大多数 continue 场景可以在第一轮逻辑审查直接发现问题
- 少数需要补查的场景通常 1–2 次工具调用足够
- 若给 Audit Agent 太大 budget，它会退化为第二个 Diagnosis Agent，破坏架构清晰性

#### Audit Agent Budget 耗尽时的默认决策

当 Audit Agent 达到 `tool_calls_limit` 或 `max_rounds` 时：
- 若已发现强反驳证据：返回 `continue`（让 Diagnosis Agent 补查）
- 若未发现强反驳证据：返回 `pass`（审查者穷尽资源仍未推翻→默认放行）
- 若假设空间整体失效：返回 `degrade`

#### 四种决策的语义

- `pass`：Audit Agent 未发现仍未解释的关键漏洞，允许 FINALIZE
- `continue`：当前证据不足以支持结论，但仍有明确可补查方向
- `rehypothesize`：现有假设空间整体失效，继续补查已无意义，需要重开假设
- `degrade`：无论补查还是重开都缺乏预算或可靠性，不再继续

#### 快路径

Audit Agent 的第一轮逻辑审查允许零工具直接结束：

- **零工具 continue**：例如只有一条 CPU 升高证据，但 memory/network 替代假设均未排除
- **零工具 pass**：例如 Diagnosis Agent 已提供三条独立证据且所有替代假设均被直接反驳

#### hint 语义规范

`hint` 只在 `decision = continue` 时必填，要求满足：

- 是可执行的问题，而不是笼统抱怨
- 指向一个或两个具体验证缺口
- 不直接包含 Audit Agent 自己的最终判断

例子：

- 好的 hint：`请验证 memory 异常是否先于 cpu 异常出现，并检查 swap 是否同步上升`
- 差的 hint：`你的结论不对，重新想`

#### gate_hint 历史策略

Audit Agent 只能在 state 中看到最近一次 `gate_hint`（如果非空）。历史 hint 不保留在 Audit Agent 可见输入中，但完整记录在 `audit_trace` 中供事后审计。THINK 同样只响应最新一条 `gate_hint`。

#### 终止条件

Audit Agent 在满足任一条件时结束：

1. 找到足以推迟放行的关键漏洞
2. 找不到任何关键漏洞
3. 已达 `tool_calls_limit`
4. 已达 `max_rounds`

#### 输出 Schema

```json
{
  "decision": "pass | continue | rehypothesize | degrade",
  "reason": "说明裁决依据",
  "hint": "下一步该验证什么；仅 continue 时非空",
  "diagnosis_type": "single_fault | composite_fault | partial | inconclusive | null"
}
```

### 4.8 OBSERVE 节点详细设计

OBSERVE 节点刻意收窄职责，不再裁决证据强弱，只做以下四件事：

1. **结果压缩**：把工具输出转成 1–2 句 `result_digest`
2. **统计量提取**：把关键数值放进 `raw_stats`
3. **verification 对齐**：标记哪些 verification item 被触达
4. **异常处理**：把工具超时、缺失数据、空结果编码成结构化状态

OBSERVE 明确不做：

- 不根据 `strength_thresholds` 打标签
- 不决定假设 confirmed 或 refuted
- 不决定是否进入 FINALIZE

这使 OBSERVE 成为一个“上下文压缩器”，而不是“半个裁判”。

### 4.9 Diagnosis-Audit 交互协议

系统采用方案 C：Audit Agent 的调查结果不直接写回 Diagnosis Agent 证据池，而是通过 `gate_hint` 对 THINK 形成最小必要约束。

#### 协议流程

1. THINK 触发 `conclude`
2. Audit Agent 独立审查并返回 `decision`
3. 若 `decision = continue`：
   - `gate_hint` 写入 state
   - `audit_evidence` 保留在独立字段
   - THINK 下一轮必须优先响应 `gate_hint`
4. 若 `decision = pass` 或 `degrade`：
   - `gate_hint = null`

#### THINK 的响应规则

THINK 读取到 `gate_hint` 时：

- 下一轮优先围绕 hint 补查
- 若 hint 指向多个缺口，可自主排序
- 完成 hint 所需验证后，才允许再次 `conclude`

#### 方案 C 的优势

- **保留角色边界**：Audit Agent 不直接替 Diagnosis Agent 做工作
- **减少重复劳动**：Diagnosis Agent 不必再次从零发现漏洞
- **便于审计**：用户能清晰看到“Audit Agent 提了什么问题，Diagnosis Agent 如何响应”

## Part V：Knowledge Base

### 5.1 两层结构

知识库分为两层，在系统启动前离线构建一次：

| 层次 | 存储格式 | 内容 |
|------|---------|------|
| **Metric KB** | YAML + ChromaDB 向量索引 | 指标语义描述、正常范围、`strength_thresholds`、`downstream_effects` |
| **Fault Pattern Library (FPL)** | JSONL | 故障模式规则，初始为空，通过 Reflect 积累 |

### 5.2 Metric KB 条目规格

```yaml
- name: "cpu_usage_percent"
  subsystem: "cpu"
  description: "面向诊断的语义描述"
  unit: "%"
  normal_range: {min, max}
  strength_thresholds:
    weak:   {condition, min_duration_sec}
    medium: {condition, min_duration_sec}
    strong: {condition, min_duration_sec}
  related_faults: ["cpu_fullload"]
  downstream_effects:
    - metric: "memory_usage_percent"
      lag_range_sec: [120, 600]
      condition: "if_sustained"
```

关键设计：

- `strength_thresholds` 不再由 OBSERVE 确定性执行，而是供 Audit Agent 作为参考文档理解“这个指标通常何时算强信号”
- `downstream_effects` 仍是复合故障甄别的重要参考，用于判断某异常是否更像继发症状
- `related_faults` 在 FPL 冷启动阶段帮助 HYPOTHESIZE 枚举 plausible fault types

### 5.3 FPL 条目规格

```json
{
  "pattern_id": "fpl_xxx",
  "fault_type": "cpu_fullload",
  "version": 1,
  "status": "active | confirmed | deprecated",
  "source": "manual | reflected",
  "provenance_exp_ids": [],
  "confidence": 0.5,
  "symptom_signature": {
    "leading_subsystem": "cpu",
    "required_metrics": [],
    "optional_metrics": [],
    "temporal_pattern": "描述时序特征",
    "causal_order_hint": []
  },
  "verification_steps": [],
  "composite_discrimination": {
    "cascade_indicator": "描述级联关系的判据",
    "composite_indicator": "描述独立复合故障的判据"
  },
  "solutions": []
}
```

> `composite_discrimination` 为可选字段：复合故障模式必填，单一故障模式可省略。

规则生命周期：

1. Reflect 生成候选规则，初始 `status = active`
2. 后续被多次命中并通过诊断确认后升级为 `confirmed`
3. 表现恶化后降级为 `deprecated`
4. deprecated 不是永久终态：若后续重新命中且诊断成功，可重新激活为 active

## Part VI：Evidence Gating

Evidence Gating 是系统的核心机制。它不是人工写死的规则函数，而是由架构级独立的 Audit Agent 执行。其目标不是“机械打分”，而是“以反方视角寻找结论漏洞”。

### 6.1 三项核心原理

**原理 1：证据独立性**

Audit Agent 会检查支持性证据是否只是同一现象的重复描述。例如，同一轮 CPU 查询产生的多个聚合统计不应被当成多条独立证据。

**原理 2：反驳优先**

一条强有力的反驳线索足以阻止通过，即使支持性证据很多。Audit Agent 会优先寻找“解释不了的反例”而不是累加支持点数。

**原理 3：复合故障甄别**

当多个假设都看似成立时，Audit Agent 需要区分真正的独立根因和下游症状链。它主要借助 `causal_order`、`downstream_effects` 与异常幅度/时滞进行判断。

### 6.2 Audit Agent 的三阶段思考框架

Audit Agent 不以 Python 三阶段规则实现，但其 system prompt 仍要求它按以下顺序思考：

```text
Stage A: 单假设审查
  - 当前结论的核心支持证据是什么？
  - 有哪些替代解释尚未排除？
  - 是否存在直接反驳线索？

Stage B: 多假设关系审查
  - 若多个假设同时成立，它们是独立故障还是因果链？
  - 哪些异常更像 derived symptoms？

Stage C: 全局裁决
  - pass / continue / rehypothesize / degrade
```

### 6.3 Audit Agent 的裁决逻辑

#### `pass`

适用条件：

- 关键替代解释已被排除
- 主要异常都有解释归属
- 不存在明显未验证的关键缺口

#### `continue`

适用条件：

- 当前结论有合理性，但仍存在可补查的关键漏洞
- 漏洞能被 1–2 个额外查询验证

`continue` 的本质不是“失败”，而是“尚不足以放行”。

#### `rehypothesize`

适用条件：

- 当前假设集合整体无法解释观测
- 继续在现有假设上补查已无明显价值
- 仍有足够预算允许重开假设空间

#### `degrade`

适用条件：

- 预算耗尽
- 数据严重缺失
- 审查发现问题，但已不足以支撑继续推进

`degrade` 不是“系统失败”，而是诚实地输出部分诊断或不确定诊断。

### 6.4 诊断结论零硬编码阈值的含义

这里的"诊断结论零硬编码阈值"并不意味着系统完全没有数值参数。系统包含两层参数：

**信号抽取层**（Triage）：Z-score 阈值、持久性比例、Top-K 等参数属于统计检测的超参数，类似于传统信号处理中的低通截频。它们不决定诊断结论。

**诊断裁决层**（Audit Agent）：不存在"满足 X 条件即通过"的人工规则。裁决由受约束的 Adversarial LLM 根据证据上下文自主判断。

数值仍然存在于 `raw_stats`、Metric KB 的 `strength_thresholds`、`downstream_effects` 中，但这些只是 Audit Agent 的审查依据，而不是外部研究者手工编写的强制裁决逻辑。

### 6.5 为什么采用对抗式 Audit Agent

若将审查机制设计为人工规则集合，系统会不可避免地落回“由研究者预设通过条件”的范式。对抗式 Audit Agent 通过角色分离、信息隔离和结构化输出，把“是否通过”的判断交给受控 LLM，同时保留审计轨迹与消融边界，更符合系统的自主诊断目标。

## Part VII：Reflect

FINALIZE 完成后按条件触发 Reflect。核心职责：将本次诊断经验提炼为 FPL 规则，实现无监督知识积累。

### 7.1 触发前置条件

Reflect 不是无条件触发，仅在有可信 confirmed 根因时才有可提炼信号：

| diagnosis_type | Reflect 策略 |
|---------------|--------------|
| `single_fault` / `composite_fault` | 完整触发，提炼所有 confirmed 假设 |
| `partial` | 部分触发，只提炼 confirmed 假设，且标记 `incomplete = true` |
| `inconclusive` | 跳过 |

### 7.2 执行步骤

1. **规则提炼**：LLM 分析 confirmed 假设的 symptom signature 与 verification path，生成候选规则
2. **去重写入**：与已有 FPL 做相似度比较，相似则更新版本，不相似则新增
3. **规则升级 / 衰减**：根据后续命中表现调整 `confidence` 和 `status`

### 7.3 FPL 冷启动行为

FPL 为空时，系统仍然可运行：

- `prior_confidence` 使用默认值 0.30
- `verification_steps` 缺省，由 THINK 自主探索
- `related_faults` 与 `causal_order` 仍可支撑初始假设生成

### 7.4 防污染措施

- LLM 只生成 schema 允许的字段
- `confidence` 由系统根据后续表现更新，不允许 LLM 自行填写最终数值
- 写入前做 Pydantic 校验

## Part VIII：评估方案

### 8.1 Ground Truth 比对（独立于 Reflect）

在实验评估模式下，使用 `chaos.csv` 作为 Ground Truth：

- 对比 `DiagnosisReport.root_causes` 与实际注入故障
- 计算 Hit@1、Hit@3、Composite Coverage
- 评估结果只用于分析，不回流进入 Reflect

### 8.2 数据集

| 数据集 | 规模 | 用途 |
|--------|------|------|
| formaltest | 29 实验 | 主评测 |
| compositetest_backup | 8 实验 | 复合故障补充评测 |

### 8.3 评估指标

| 类别 | 指标 | 说明 |
|------|------|------|
| 准确性 | Root Cause Hit@1 | Top-1 根因匹配率 |
| 准确性 | Root Cause Hit@3 | Top-3 覆盖率 |
| 准确性 | Composite Coverage | 复合故障完整诊断比例 |
| 效率 | Avg Tool Calls | 平均工具调用次数 |
| 效率 | Avg Audit Tool Calls | Audit Agent 平均额外工具调用次数 |
| 效率 | End-to-end Latency | 总耗时 |
| 成本 | Token Usage | Diagnosis Agent / Audit Agent / FINALIZE 分项 token |
| 鲁棒性 | Hit@1 @ 20% missing | 缺失指标下性能退化 |
| KB 增益 | ΔHit@1 (with/without FPL) | 知识积累收益 |
| Audit Agent 质量 | True Pass / False Pass / False Reject | Audit Agent 放行后诊断正确率 vs Audit Agent 拦截后最终诊断正确率 |

### 8.4 跨模型评估

在主要评测模型（DeepSeek-Chat）之外，至少使用 2 个额外模型（如 GPT-4o-mini、Qwen 系列）做全量评测，评估系统对底层 LLM 的依赖程度。特别关注：
- Diagnosis Agent 和 Audit Agent 使用同一模型 vs 不同模型时的诊断差异
- 开源模型（如 Llama/Qwen）在 Audit Agent 审查任务上的可靠性

## Part XI：失败处理

| 场景 | 处理策略 |
|------|---------|
| 主工具超时 | 生成不可用证据，禁止作为支持性结论依据 |
| Audit Agent 工具超时 | Audit Agent 可选择 `continue` 或 `degrade`，不得凭空补完证据 |
| 工具返回空数据 | 对应 verification 标记为 `unverifiable` |
| THINK 输出不符合 Schema | 最多重试 2 次，仍失败则降级输出 |
| Audit Agent 输出不符合 Schema | 最多重试 2 次，仍失败则退化为 `degrade` |
| Budget 耗尽且已有 confirmed | 输出 `partial` |
| Budget 耗尽且无 confirmed | 输出 `inconclusive` |
| 全部假设 refuted 且可重开 | 触发 `rehypothesize` |
| 全部假设 refuted 且不可重开 | `degrade` |
| FPL 命中但验证失败 | 规则只作为先验，不允许直接下结论 |
| LLM API 故障（限流/500/超时） | 重试 2 次，仍失败则 degrade |
| HYPOTHESIZE 生成 0 个假设 | 重试 1 次；仍失败则 degrade + 仅输出 Triage 结果 |
| Triage 检测到 0 个异常指标 | 直接输出 `no_anomaly_detected`，跳过 Stage 2 |
| THINK 不 conclude 循环（超过 max_react_iterations） | 设默认 max_react_iterations=15，超过后强制 conclude |
| FPL Reflect 写回失败（IO 错误） | 记录 warning，不影响主诊断输出 |
| metrics.csv 格式异常或严重缺列 | 入口校验，退化为 best-effort 模式 |

| 阶段 | 内容 | 关键产物 |
|------|------|---------|
| Phase A | Metric KB + FPL 初始构建 | `kb/metrics.yaml`, `kb/fpl.jsonl` |

Phase C 的实现建议拆成三步：

1. 先实现主图（HYPOTHESIZE / THINK / ACT / OBSERVE / FINALIZE）
2. 再加入最小 Audit Agent（单轮、零工具）+ Orchestrator 验证路由
3. 最后扩展成完整 Audit Agent（含工具调用与方案 C 的 hint 协议）

## Part XIII：一次完整诊断的控制流示例

为了让实现与论文叙事统一，推荐在附录或实现文档中固定使用如下控制流叙事：

1. Triage 生成 `FocusContext`
2. HYPOTHESIZE 生成 2–4 个候选假设
3. THINK 进入主 ReAct 循环，逐步收集 supporting / refuting evidence
4. THINK 在认为证据充分时触发 `conclude`
5. Orchestrator 将 ConclusionProposal 提交给 Audit Agent
6. Audit Agent 先做零工具逻辑审查；若发现缺口，返回 `continue + hint`
7. THINK 根据 `gate_hint` 继续补查
8. 若当前假设空间失效，Audit Agent 返回 `rehypothesize`
9. 若找不到漏洞，Audit Agent 返回 `pass`
10. FINALIZE 生成 DiagnosisReport
11. Reflect 在符合条件时回写 FPL

这个控制流显式展示了“诊断者提出主张，审查者独立质疑，再决定是否放行”的闭环，是系统控制逻辑的核心叙事。
