# HPC-Diagnosis Agent Design (v1.1)

## 1. Architecture Overview

本设计参考 SC25 Stellar 论文，将 HPC 异常诊断从简单的 "LLM 调用" 升级为具备深度推理能力的 **Agentic Workflow**。
系统采用 **三层分层架构**，结合 **确定性计算 (Deterministic Computation)** 与 **概率性推理 (Probabilistic Reasoning)**，实现高效、精准、可进化的诊断。

架构图参考: `design/draw/v1.drawio`

### Core Layers

*   **Layer 0: Knowledge Base (Offline Preparation)**
    *   系统的“长期记忆”与“先验知识”。
    *   为 Agent 提供指标语义、基线统计特征及历史故障模式。
*   **Layer 1: Triage Agent (Deterministic Workflow)**
    *   系统的“分诊台”。
    *   使用低成本、高可靠的统计算法快速圈定异常范围，过滤噪声。
*   **Layer 2: Diagnosis Agent (LangGraph Reasoning Loop)**
    *   系统的“专家医生”。
    *   基于 LLM 的自主智能体，通过“假设-验证”循环进行深度根因分析。
*   **Layer 3: Evolution (Reflect & Accumulate)**
    *   系统的“进化机制”。
    *   通过诊断后的反思，将新案例提炼为规则，不断扩充故障模式库。

## 2. Detailed Component Design

### Layer 0: Knowledge Base
| 组件 | 描述 | 实现技术 |
|------|------|----------|
| **Metric KB** | 指标知识库。包含 150+ Prometheus 指标的物理含义、单位、正常阈值范围、相关子系统。 | ChromaDB (Vector) + YAML |
| **Fault Pattern Library** | 故障模式库。存储“症状签名 -> 根因”的映射规则。来源于历史 Few-shot 及 Layer 3 的自动提炼。 | ChromaDB / JSON |
| **Baseline Profile** | 基线画像。系统在无故障运行时的统计特征（均值、方差、周期性），用于 Layer 1 的异常判定。 | Pandas / Scipy |

### Layer 1: Triage Agent (The Filter)
*   **输入**: 原始 Prometheus Metrics (TimeSeries), Job Logs
*   **逻辑**: 
    1.  **Anomaly Detection**: 对关键指标（CPU/Mem/IB/Lustre）应用 Z-Score 或固定阈值检测。
    2.  **Job Association**: 将异常时间窗口映射到具体作业（Job ID）。
    3.  **Noise Reduction**: 过滤瞬时抖动，仅保留持续性异常。
*   **输出**: `Triage Report` (JSON)
    *   异常时间段 (Start/End)
    *   受影响节点列表
    *   Top-K 异常指标及其偏离度
*   **设计原则**: **Reliability over Flexibility**. 不使用 LLM，确保 100% 可复现且零 Token 成本。

### Layer 2: Diagnosis Agent (The Reasoner)
*   **架构**: 基于 **LangGraph** 的 StateGraph。
*   **状态流转 (The Loop)**:
    1.  **Hypothesize (假设)**: 基于 Triage Report 和 Fault Pattern Library，提出可能的根因假设（例如：“怀疑是 Lustre OST 拥塞”）。
    2.  **Tool Selection (规划)**: 决定需要查阅什么额外信息来验证假设。
    3.  **Execution (执行)**: 调用工具获取证据。
    4.  **Verify (验证)**: 分析工具返回的数据，确认或排除假设。
    5.  **Decide (决策)**: 证据是否确凿？是 -> 生成报告；否 -> 回到 Hypothesize。
*   **工具箱 (Tools)**:
    *   `MetricQueryTool`: 查询指定时间段、指定节点的细粒度指标。
    *   `KBRetrievalTool`: 查询指标含义或相关故障案例。
    *   `DataAnalysisTool`: 执行相关性分析等统计任务。

### Layer 3: Evolution (The Learner)
*   **触发时机**: 诊断结束，生成最终报告后。
*   **逻辑**: 
    1.  **Review**: 将本次诊断的“症状+根因”与库中已有规则对比。
    2.  **Extract**: 如果是新模式，提炼为一条通用规则（Rule Extraction）。
    3.  **Update**: 将新规则存入 Fault Pattern Library。

## 3. Implementation Plan

### Phase A: Foundation (Layer 0)
*   [ ] 构建 `Metric KB`: 整理 `metrics.yaml`，补充指标描述，存入 ChromaDB。
*   [ ] 构建 `Fault Pattern Library`: 从现有 few-shot txt 中提取规则，结构化存储。
*   [ ] 建立 `Baseline Profile`: 跑通无故障 workload，生成基线统计数据。

### Phase B: Triage Pipeline (Layer 1)
*   [ ] 实现 `AnomalyDetector`: Python 脚本，基于统计规则筛选异常。
*   [ ] 实现 `JobAssociator`: 关联 metrics 与 jobinfo。
*   [ ] 输出标准化的 `Triage Report`。

### Phase C: Diagnosis Agent (Layer 2)
*   [ ] Setup **LangGraph**: 定义 State, Nodes, Edges。
*   [ ] 实现 Tools: `MetricQuery`, `KBRetrieval`。
*   [ ] Prompt Engineering: 编写 System Prompt，引导 Agent 进行思维链推理 (CoT)。

### Phase D: Evolution Loop (Layer 3)
*   [ ] 实现 `Reflector`: 诊断后的自动复盘模块。
*   [ ] 实现规则合并逻辑。

## 4. Key Differences from Baseline
| 特性 | 旧版 (Current Baseline) | 新版 (Agentic Design) |
|------|-------------------------|-----------------------|
| **输入处理** | 一次性拼接全量数据 (Context Window 压力大) | Triage 先过滤，Agent 按需查询 (Token 效率高) |
| **推理模式** | 单次问答 (One-shot) | 多步迭代 (ReAct Loop) |
| **知识利用** | 仅靠 LLM 预训练知识 + Few-shot | RAG (外部知识库) + 动态检索 |
| **可解释性** | 弱 (黑盒输出) | 强 (完整的推理步骤记录) |
| **成长性** | 无 | 有 (Fault Pattern Library 持续积累) |
