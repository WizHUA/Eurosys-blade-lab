# HPC-Diagnosis Agent Design (v3)

本版本将原先独立的 Layer 1 (Triage Agent) 合并进 Diagnosis Agent，使系统以单一 Agent 形式运行：同一个 Agent 内部先执行确定性分诊（降维/降噪/聚焦），再进入假设-验证循环完成根因定位与建议生成。

## 1. Architecture Overview

### 1.1 Layer 0：Offline Knowledge Base（长期记忆）

- **Metric KB**：指标语义/单位/典型范围/子系统归属/常见关联故障。
- **Baseline Profile**：无故障/稳态下的统计特征（均值、方差、分位数、周期性）。
- **Fault Pattern Library**：症状签名 → 根因/置信度/验证策略/处置建议（可增量更新）。

### 1.2 Single Diagnosis Agent（内置分诊 + 推理闭环）

输入仍为 `metrics.csv + jobinfo.csv (+ 可选 chaos.csv 作为离线评估 ground truth)`，但处理被统一到一个 Agent 内：

1) **Deterministic Triage（Agent 内部的确定性分诊阶段）**
- 目标：将“原始时序”压缩为“可推理的聚焦上下文（Focus Context）”。
- 产物：Top-K 异常指标、异常窗口、节点/作业映射、异常强度排序、可疑子系统初判。

2) **Reasoning Loop（Hypothesize → Gather → Verify → Finalize）**
- 目标：在聚焦上下文上进行可控的假设-证据闭环，输出结构化诊断结论。

### 1.3 Layer 3：Evolution（反思与积累）

- 诊断完成后进行复盘：将本次“症状→根因→有效证据”提炼为规则条目，写回 Fault Pattern Library。
- 该模块可以作为 Agent 的终止节点（同一进程内完成），也可作为异步离线任务运行。

## 2. Diagnosis Agent 内部拆解

### 2.1 内置分诊（Deterministic Triage）的具体职责

- **异常检测**：基于 Baseline Profile 做 Z-score/分位数偏离/静态阈值的组合判定。
- **噪声抑制**：时间窗口持久性过滤（例如连续 N 个采样点超阈值）与峰值剪裁。
- **作业关联**：按时间重叠与节点集合映射异常窗口 ↔ 作业。
- **聚焦表示**：输出 Focus Context（用于后续 Prompt/工具调用的短上下文）。

分诊被合并后，不再作为独立 Agent 对外暴露，而是 Diagnosis Agent 的首个确定性节点/工具调用，保证一致性与可维护性。

### 2.2 推理闭环（Reasoning Loop）

Agent 在 Focus Context 上执行：

- **Hypothesize**：结合 Fault Pattern Library 给出候选根因集合（带优先级/置信度/需要的证据列表）。
- **Gather**：按需调用工具补充证据（而不是回看全量 metrics）。
- **Verify**：对证据进行一致性检验与反证（排除假设/收敛假设）。
- **Finalize**：输出诊断报告（根因、证据、建议、剩余不确定性）。

## 3. Tools（Agent 的“可执行能力”）

合并分诊后，Diagnosis Agent 的工具箱建议包含 4 类（其中 1 类是“内置分诊工具/节点”）：

1) **TriageTool（确定性分诊）**
- 输入：原始 metrics/jobinfo。
- 输出：Focus Context（异常窗口、Top-K 指标、节点/作业映射、异常排序）。

2) **MetricQueryTool（按需查数）**
- 查询“特定节点 + 特定时间窗口 + 特定指标”的细粒度数据，用于验证假设。

3) **KBRetrievalTool（检索先验）**
- 从 Metric KB / Fault Pattern Library 检索指标语义、典型故障签名、验证路径。

4) **DataAnalysisTool（统计分析）**
- 相关性/滞后相关、变化点检测、分组对比（异常节点 vs 正常节点）、聚合与特征提取。

## 4. Why This Change（为何合并 Layer 1）

- **工程上更贴合任务**：分诊逻辑简单且确定，不值得独立成“Agent 边界”；放入同一 Agent 降低组件间协议/状态同步成本。
- **推理上更可控**：Agent 在同一 state 内获得分诊产物，避免跨 Agent 传递导致的上下文损失与格式漂移。
- **实现上更直接**：LangGraph 可将分诊作为第一个节点（纯 Python 计算），后续节点再进入 LLM + Tools 的闭环。

---

## 5. Implementation Plan（v3）

- **Phase A**：构建/补全 Metric KB、Baseline Profile、Fault Pattern Library。
- **Phase B**：实现 TriageTool（确定性分诊）并与 Agent state 对接。
- **Phase C**：实现 Reasoning Loop（LangGraph），接入 MetricQuery/KBRetrieval/DataAnalysis。
- **Phase D**：实现 Reflect & Accumulate，将新规则写回 Fault Pattern Library。
