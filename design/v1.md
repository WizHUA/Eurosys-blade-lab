## 一、现状与机会分析报告

### 1.1 当前系统核心工作流

当前系统是一个**两阶段串行管线**：

- **Phase 1 (dataset_builder)**：`ExperimentManager` 编排三个组件并发运行 — `PrometheusMonitor`(采集150+指标)、`ChaosManager`(注入7种故障/21种组合)、`WorkloadManager`(提交NPB作业)。产出 `metrics.csv` + `jobinfo.csv` + `chaos.csv`。
- **Phase 2 (llm)**：将 system prompt + 3~5个 few-shot + **全量 metrics+jobinfo 一次性拼接** → 单次 LLM 调用 → 输出 `[{anomaly, root_causes, solutions}]` JSON。

**核心局限**：整个诊断是一个 **"单次 prompt → 单次 response"** 的管道，没有推理循环、没有知识积累、没有分层分析。

### 1.2 STELLAR 最值得引入的 5 个设计模式

| # | STELLAR 机制 | 论文章节 | 我们可解决的痛点 |
|---|-------------|---------|----------------|
| **D1** | RAG 驱动的离线知识准备 | §4.2 | **P5** 缺乏指标语义知识 — 为150+指标建立含义、正常范围、故障关联的知识库 |
| **D2** | 双 Agent 分层推理 (Analysis→Tuning) | §4.3 | **P1,P2** 推理浅且复合故障差 — Triage Agent 降噪+聚焦，Diagnosis Agent 深度推理 |
| **D3** | 反思 & 规则集积累 (Reflect→Rule Set) | §4.4 | **P4** 经验无法积累 — 每次诊断后提炼故障模式规则，加速未来诊断 |
| **D4** | 自主工具调用循环 (Tool Use Loop) | §4.3.2 | **P1,P6** 推理浅+无法交互 — Agent可自主选择查更多指标/检索知识/运行分析 |
| **D5** | 结构化推理链记录 (Documented Rationale) | §4.4.1 | **P3** 不可解释 — 每步推理记录依据，形成可追溯诊断链 |

## 二、HPC-Diagnosis Agent 架构设计

### 2.1 总体架构（三层管线）

```
═══════════════════════════════════════════════════════
  Layer 0: Offline Preparation (一次性构建，持续更新)
═══════════════════════════════════════════════════════
  ┌─────────────────┐  ┌──────────────────┐  ┌──────────────────┐
  │ Metric KB       │  │ Fault Pattern    │  │ Normal Baseline  │
  │ (指标知识库)     │  │ Library (规则集)  │  │ Profile (基线)   │
  │                 │  │                  │  │                  │
  │ 指标语义/范围    │  │ 故障签名→根因    │  │ 无故障时的统计   │
  │ 因果关联图谱    │  │ few-shot结构化   │  │ 分布特征        │
  │ ChromaDB 存储   │  │ JSON 持久化      │  │                  │
  └─────────────────┘  └──────────────────┘  └──────────────────┘

═══════════════════════════════════════════════════════
  Layer 1: Triage Agent (确定性 Workflow, 非LLM)
═══════════════════════════════════════════════════════
  metrics.csv + jobinfo.csv
       ↓
  [统计异常检测] → [资源维度分组] → [时间窗口聚合]
       ↓
  Triage Report (只含异常段落 + 关联作业 + 异常摘要)

═══════════════════════════════════════════════════════
  Layer 2: Diagnosis Agent (LangGraph Agent, 自主推理)
═══════════════════════════════════════════════════════
  Triage Report + 规则集
       ↓
  ┌──────────────────────────────────────────────┐
  │  假设生成 ←→ 证据收集 ←→ 假设验证  (循环)    │
  │              │                               │
  │         ┌────┴────┐                          │
  │    ┌────┴──┐ ┌────┴────┐ ┌─────────┐        │
  │    │Metric │ │Knowledge│ │  Stat   │        │
  │    │Query  │ │Retrieval│ │Analysis │        │
  │    │Tool   │ │Tool     │ │Tool     │        │
  │    └───────┘ └─────────┘ └─────────┘        │
  └──────────────────────────────────────────────┘
       ↓
  结构化诊断报告 + 完整推理链

═══════════════════════════════════════════════════════
  Layer 3: Post-Diagnosis (Reflect & Accumulate)
═══════════════════════════════════════════════════════
  [对比 chaos.csv ground truth] → [提炼新规则] → [合并规则集]
```

### 2.2 STELLAR → HPC 诊断：模式映射

| STELLAR 概念 | 我们的等价物 | 差异说明 |
|-------------|-------------|---------|
| Analysis Agent (分析 Darshan 日志) | **Triage Agent** (分析 Prometheus 指标) | 我们用确定性 Workflow 而非 LLM，因为阈值检测更可靠 |
| Tuning Agent (试错循环) | **Diagnosis Agent** (假设-验证循环) | STELLAR 是"配置→执行→验证"，我们是"假设→查证→确认" |
| RAG-based Parameter Extraction | **Metric Knowledge Base** | STELLAR 从手册提取参数，我们从 metrics.yaml + LLM 标注提取指标知识 |
| Rule Set Accumulation | **Fault Pattern Library** | STELLAR 积累调优规则，我们积累故障诊断规则 |
| Configuration Runner Tool | **MetricQueryTool** | STELLAR 跑应用验证，我们查指标验证假设 |

### 2.3 Triage Agent — 为什么用 Workflow 而非 Agent？

参考 STELLAR §4.3.3 的讨论："Workflows prioritize reliability at the cost of flexibility"。异常检测是确定性任务（对比基线阈值），**不需要 LLM 的灵活性**。用确定性代码保证：
- 100% 可复现（同样的输入一定产生同样的 Triage Report）
- 零 token 消耗（省 LLM 成本给 Diagnosis Agent）
- 150+ 指标快速筛选到 5~15 个关键异常

### 2.4 Diagnosis Agent — 工具调用循环

LangGraph `StateGraph` 实现，核心状态转移：

```
                ┌──────────────┐
       ┌────→   │  HYPOTHESIZE │ ←──────┐
       │        │  提出假设     │        │
       │        └──────┬───────┘        │
       │               ↓                │
       │        ┌──────────────┐        │
       │        │   GATHER     │        │ 假设全部排除 →
       │        │   收集证据    │        │ 重新假设
       │        │  (调用Tools)  │        │
       │        └──────┬───────┘        │
       │               ↓                │
       │        ┌──────────────┐        │
       │        │   VERIFY     │────────┘
       │        │   验证假设    │
       │        └──────┬───────┘
       │               ↓
       │        ┌──────────────┐
       └── no ──│  FINALIZE?   │
                │  足够诊断?    │
                └──────┬───────┘
                       ↓ yes
                ┌──────────────┐
                │   OUTPUT     │
                │  输出报告     │
                └──────────────┘
```

### 2.5 技术选型

| 组件 | 选择 | 理由 |
|------|------|------|
| Agent 框架 | **LangGraph** | 你的偏好; StateGraph 天然适合推理循环; 比 AutoGen 更轻量 |
| 向量数据库 | **ChromaDB** | 纯 Python、零运维、适合小规模知识库 |
| LLM API | **OpenRouter** (保持现有) | 兼容多模型，无需改造 |
| 统计分析 | **pandas + scipy** | 已在项目中，无新依赖 |

### 2.6 实现分阶段路线

| 阶段 | 内容 | 依赖 |
|------|------|------|
| **Phase A** | 离线准备：从 metrics.yaml 构建 Metric KB；从 few-shot 构建 Fault Pattern Library；基线 Profile | 无 |
| **Phase B** | Triage Agent：统计异常检测 + Triage Report 生成 | Phase A |
| **Phase C** | Diagnosis Agent：LangGraph StateGraph + 3 个 Tools + 推理链输出 | Phase A, B |
| **Phase D** | 反思 & 积累：ground truth 对比 + 规则提炼 + 规则合并 | Phase C |

### 2.7 范围边界

- **包含**：离线诊断、单节点故障、基于 Prometheus 指标、LangChain/LangGraph
- **排除**：实时在线诊断(v2)、多节点关联(v2)、自动修复/调参、从外部文档构建 RAG
