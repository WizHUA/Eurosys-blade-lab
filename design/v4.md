# HPC-Diagnosis Agent Design (v4)

本文在 v3/v3_detail 的基础上，先系统评判其不足，再给出完整的 v4 设计方案。

---

## Part I：v3 设计评判

### 1.1 v3 的核心贡献（值得保留）

| 机制 | 评价 |
|------|------|
| 确定性分诊先于 LLM | 正确的第一性原理：降噪、压缩上下文、隔离非确定性 |
| LangGraph 状态机 | 工程上合理，可复现、可审计 |
| Evidence Gating | 论文贡献亮点，解决"编故事式推理"问题 |
| 结构化 State / DiagnosisReport | 是系统可评估的基础，必须保留 |
| Reflect & Accumulate | 与 STELLAR 对齐，知识积累是长期价值 |

### 1.2 v3 的主要问题

#### 问题 1：推理机制不明（最大缺陷）

v3 定义了 HYPOTHESIZE → GATHER → VERIFY 的三步，但没有说明**每步的推理形式**是什么。在 Agent 工程里，推理机制的选择至关重要：

- **ReAct（Reason+Act 交织）**：LLM 在每次行动前输出 Thought，每次工具返回后再推理。适合需要根据中间结果动态调整策略的场景。
- **Plan-then-Execute**：先生成全套行动计划，再批量执行工具。适合已知验证路径、步骤之间无依赖的场景。

v3 的三步结构更像 Plan-then-Execute，但 HPC 诊断的验证路径常依赖中间发现（例如，发现 CPU 异常后才决定是否查网络）。应当采用 **ReAct 交织模式**，每次工具调用前都有 Thought 输出。

#### 问题 2：单假设收敛模型不适合复合故障

v3 的 GATE 条件（Rule A/B/C）隐含了"收敛到单一根因"的模型，但实验数据中 29+ 个场景里有大量复合故障（CPU+内存、CPU+网络等）。复合故障需要**同时维护多个假设流**，不能靠单一置信度收敛来处理。

#### 问题 3：知识库实现缺失

- v3 提到 Fault Pattern Library"可增量更新"，但没有定义 Schema。
- v1/v2 中出现的 ChromaDB 在 v3 中消失，向量检索 vs 规则匹配的边界模糊。
- Baseline Profile 如何构建、如何处理不同 workload 下的基线差异，均未说明。

#### 问题 4：合并分诊削弱了消融实验的可写性

v3 把 Triage 合并为 Diagnosis Agent 的内部节点，理由是"工程上更简单"。但这同时消除了一个最重要的消融维度：**无分诊（直接输入全量 metrics）vs 有分诊**。两者如果在同一个 LangGraph 图内，消融时边界不清晰。

v4 保持两者在逻辑上的分离（两个独立的 Stage），但在同一 Python 进程内实现，兼顾工程简洁性与消融可写性。

#### 问题 5：时序因果信息被丢弃

分诊只输出"哪些指标异常"，没有输出"哪个指标先异常"。HPC 故障的根因通常是链式传播的（CPU 压力 → 内存换页 → 磁盘 IO 飙升），时序因果顺序是区分根因与症状的关键证据。

#### 问题 6：工具输出未经压缩即进入上下文

MetricQueryTool 可以返回原始时序序列，若 Agent 进行 10 次工具调用，累积 context 会超出 token 预算。v3 提到"预算"但没有说明如何压缩工具输出。

#### 问题 7：Reflect 节点机制过于模糊

"提炼新规则写回 Fault Pattern Library"没有定义：规则的格式是什么、去重/合并逻辑如何工作、LLM 提炼规则时如何防止规则污染。

## Part II：v4 设计

### 2.1 设计原则

1. **两阶段清晰分离**（Triage Stage + Agent Stage），逻辑独立、消融边界明确。
2. **ReAct 推理循环**：每次工具调用前有 Thought，每次观察后更新假设集合。
3. **多假设并行跟踪**：状态中显式维护 N 个候选假设，各自独立积累证据。
4. **时序因果先验**：Triage 输出的 Focus Context 包含指标异常的时序顺序，作为假设生成的先验。
5. **工具输出摘要化**：工具返回结果经过摘要层压缩后再写入上下文，防止 token 爆炸。
6. **规则库有 Schema**：Fault Pattern Library 有固定 JSON Schema，LLM 只能按格式写入，系统验证合法性。

### 2.2 系统架构

```
输入：metrics.csv + jobinfo.csv (+ 可选 chaos.csv 用于评估)
                │
                ▼
┌─────────────────────────────────────────┐
│  Stage 0: Offline Knowledge Base        │  ← 离线一次性构建
│  ┌─────────────┐ ┌──────────────────┐   │
│  │  Metric KB  │ │ Fault Pattern    │   │
│  │  (YAML+向量) │ │ Library (JSON)   │   │
│  └─────────────┘ └──────────────────┘   │
```

## Part III：Stage 1 — Triage（详细规格）

### 3.1 职责与输出

Triage 是纯确定性 Python 代码，不调用 LLM，不消耗 token。

**处理流程：**

```
1. 异常检测：对每个指标按列做 Z-score / 分位数偏离 / 静态阈值组合判定
2. 持久性过滤：连续超阈值比例 > persistence_ratio 才保留（去除毛刺）
3. 时序排序：记录每个异常指标首次超阈时间 t_onset，按 t_onset 排序
4. 变化点检测：对 Top-K 异常指标做 PELT 或 BOCPD，标注变化点时刻
5. 子系统分组：按指标的 subsystem 字段（来自 Metric KB）分组，计算子系统异常分数
6. 作业关联：时间窗口与 jobinfo 做交集映射
```

### 3.2 FocusContext Schema

```json
{
  "run_id": "string",
  "anomaly_window": {"start": "ISO8601", "end": "ISO8601"},
  "top_metrics": [
    {
      "metric": "string",
      "subsystem": "string",
      "direction": "+|-",
      "score": 0.0,
      "t_onset": "ISO8601",
      "onset_rank": 1,
      "change_point": "ISO8601|null"
    }
  ],
  "causal_order": ["metric_A", "metric_B", "metric_C"],
  "subsystem_scores": {"cpu": 0.0, "memory": 0.0, "network": 0.0, "disk": 0.0},
  "leading_subsystem": "string",
  "nodes": [{"node": "string", "severity": "high|medium|low"}],
  "jobs": [{"job_id": "string", "overlap_ratio": 0.0, "node_set": []}],
  "triage_confidence": 0.0
}
```

关键新增字段：
- `causal_order`：按 t_onset 排序的指标列表，是时序因果先验
- `leading_subsystem`：分诊给出的"最可能的首发子系统"初判
- `change_point`：每个指标的统计变化点，而非仅仅异常开始时刻

## Part IV：Stage 2 — Diagnosis Agent（详细规格）

### 4.1 完整 State Schema

```python
class AgentState(TypedDict):
    # 基础信息
    run_id: str
    inputs: dict                    # {metrics_path, jobinfo_path}

    # Stage 1 产物
    focus_context: FocusContext

    # 假设集合（多假设并行）
    hypotheses: list[Hypothesis]    # 见 4.2

    # 证据库（所有工具返回的摘要）
    evidence: list[Evidence]        # 见 4.3

    # ReAct 轨迹（完整 Thought-Action-Observation 序列）
    react_trace: list[ReActStep]    # 见 4.4

    # 预算追踪
    budget: dict                    # {tool_calls_used, tool_calls_limit, tokens_in, tokens_out}

    # 门控决策
    gate_result: dict               # {passed: bool, reason: str, active_hypotheses: list}

    # 最终输出
    report: DiagnosisReport | None
```

### 4.2 Hypothesis Schema

```json
{
  "id": "h1",
  "root_cause": "CPU fullload on node blade01",
  "subsystem": "cpu",
  "prior_confidence": 0.7,
  "current_confidence": 0.0,
  "supporting_evidence_ids": [],
  "refuting_evidence_ids": [],
  "required_verifications": [
    {"description": "confirm cpu_usage > 95% sustained", "verified": false},
    {"description": "confirm no co-occurring memory anomaly", "verified": false}
  ],
  "status": "active|confirmed|refuted"
}
```

设计要点：
- `prior_confidence` 由 KBRetrievalTool 命中的规则给出（规则库先验），无命中则为默认值 0.3
- `required_verifications` 由 HYPOTHESIZE 节点根据规则库的 `verification_steps` 字段生成
- `status` 由 GATE 节点（纯 Python）在每次 React 循环后更新，不由 LLM 决定

### 4.3 Evidence Schema

```json
{
  "id": "e1",
  "hypothesis_ids": ["h1"],
  "type": "supporting|refuting|neutral",
  "source_tool": "MetricQueryTool",
  "query_summary": "cpu_usage@blade01, 14:00-14:10, p95",
  "result_digest": "p95=98.7%, sustained for 8 min",
  "raw_stats": {"p95": 98.7, "duration_min": 8},
  "strength": "strong|medium|weak",
  "strength_basis": "规则：cpu_usage p95>95% 持续>5min → strong"
}
```

关键设计：`strength` 由**确定性规则**判定（来自 Metric KB 中每个指标的 `strength_thresholds` 字段），不由 LLM 主观判断。这是 Evidence Gating 可信的前提。

### 4.4 ReAct Loop 节点设计

#### 节点图（LangGraph）

```
TRIAGE_RESULT
    │
    ▼
HYPOTHESIZE ──────────────────────────────────────────────┐
    │                                                       │
    ▼                                                       │
┌─────────────────────────────────────────────────────┐    │
│  REACT_LOOP                                          │    │
│                                                      │    │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐       │    │
│  │  THINK   │──→ │   ACT    │──→ │ OBSERVE  │──┐    │    │
│  │ (LLM)   │    │(工具执行) │    │(摘要压缩) │  │    │    │
│  └──────────┘    └──────────┘    └──────────┘  │    │    │
│       ↑                                         │    │    │
│       └─────────────────────────────────────────┘    │    │
│                                                      │    │
└─────────────────────────────────────────────────────┘    │
    │                                                       │
    ▼                                                       │
  GATE ──── fail + budget ok ──────────────────────────────┘
    │
    │ pass
    ▼
FINALIZE (LLM)
    │
    ▼
REFLECT (LLM + 规则校验)
```

#### HYPOTHESIZE 节点（LLM）

输入：FocusContext + Fault Pattern Library 检索结果（Top-3 规则命中）

LLM 任务：
1. 基于 `causal_order` 和 `leading_subsystem` 提出 2-4 个候选假设
2. 对每个假设从 FPL 中填充 `required_verifications`
3. 为每个假设生成首个工具调用计划（ACT 的输入）

输出格式（强制 Pydantic 校验）：
```json
{
  "thought": "causal_order 显示 cpu_usage 最先异常，领先 memory_usage 约 2 分钟...",
  "hypotheses": [...],
  "first_action": {"tool": "MetricQueryTool", "args": {...}}
}
```

#### THINK 节点（ReAct 核心，LLM）

输入：当前 hypotheses + evidence（摘要形式）+ 最近一次 OBSERVE 结果

LLM 输出严格格式：
```json
{
  "thought": "e1 显示 CPU p95=98.7% 持续 8 分钟，h1 的 verification[0] 已满足...",
  "action": {
    "type": "tool_call|conclude|escalate",
    "tool": "MetricQueryTool",
    "args": {...},
    "reasoning": "需要验证 h1 的 verification[1]：是否无共发内存异常"
  }
}
```

#### OBSERVE 节点（确定性 Python）

接收工具原始输出，执行：
1. 统计摘要（均值、p95、max、持续时长等）
2. 对照 Metric KB 中的 `strength_thresholds` 打 `strength` 标签
3. 对照当前 hypotheses 更新每个假设的 `required_verifications` 状态
4. 写入 `evidence` 列表（只写 `result_digest` + `raw_stats`，不写原始序列）

#### GATE 节点（纯 Python 规则判断）

```python
def gate(state: AgentState) -> Literal["pass", "continue", "degrade"]:
    active = [h for h in state.hypotheses if h.status == "active"]
    confirmed = [h for h in state.hypotheses if h.status == "confirmed"]

    # 通过条件（满足其一）
    for h in confirmed:
        evidences = [e for e in state.evidence if h.id in e.hypothesis_ids]
        supporting = [e for e in evidences if e.type == "supporting"]
        # Rule A：2条独立支持证据，至少1条strong（独立 = 来自不同指标/不同工具）
        if len(set(e.source_tool for e in supporting)) >= 2 and \
           any(e.strength == "strong" for e in supporting):
            return "pass"
        # Rule B：1条strong + 所有required_verifications已完成
        if any(e.strength == "strong" for e in supporting) and \
           all(v.verified for v in h.required_verifications):
            return "pass"
        # Rule C：FPL规则命中（prior_confidence > 0.8）+ 最小验证通过
        if h.prior_confidence > 0.8 and any(v.verified for v in h.required_verifications):
            return "pass"

    # 预算耗尽
    if state.budget["tool_calls_used"] >= state.budget["tool_calls_limit"]:
        return "degrade"

    return "continue"
```

注意：证据"独立性"定义为**来自不同工具调用且针对不同指标**，由 `source_tool` + `query_summary` 的组合哈希去重验证，不由 LLM 判断。

### 4.5 复合故障支持

v4 显式支持多根因并发的场景：

1. HYPOTHESIZE 可以生成多个**互不排斥**的假设（如 h1=CPU故障, h2=内存故障），两者 status 可以同时进入 confirmed。
2. GATE 对所有 confirmed 假设分别判断，全部通过后才触发 FINALIZE。
3. FINALIZE 的 `root_causes` 字段支持多条目，每条独立附带置信度与证据链。
4. 特殊情况：若 h2 的异常被发现是 h1 导致的（因果链），则 h2 降级为 symptom（状态改为 `derived`），不作为独立根因输出。

## Part V：Tools（完整工具契约）

### 5.1 TriageTool（内部，不对 LLM 暴露）

已在 Stage 1 作为确定性 Python 函数运行，结果以 FocusContext 形式注入 State，不作为 LLM function call 工具。

### 5.2 MetricQueryTool

```
输入 Schema：
  nodes: list[str]               # 节点名列表
  metrics: list[str]             # 指标名列表（必须在 Metric KB 中存在）
  time_window: {start, end}      # ISO8601
  aggregation: raw|mean|p95|max|rate|duration_above_threshold
  threshold: float (仅当 aggregation=duration_above_threshold 时)

输出 Schema（摘要形式，非原始序列）：
  query_id: str
  results: [{node, metric, aggregation_type, value, unit}]
  missing: [{node, metric, reason}]
  time_window: {start, end}

注：raw 模式仅在 debug 时使用，不进入 LLM context；
    LLM 接收的永远是 aggregation 后的摘要
```

### 5.3 KBRetrievalTool

```
输入 Schema（两种模式）：
  mode: "metric_lookup" | "pattern_match"

  # metric_lookup 模式
  metric_names: list[str]

  # pattern_match 模式
  symptom: {
    subsystem: str,
    anomaly_metrics: list[str],
    causal_order_hint: list[str]  # 来自FocusContext.causal_order
  }

输出 Schema：
  # metric_lookup
  metric_entries: [{name, description, unit, normal_range, subsystem,
                    strength_thresholds, related_faults}]

  # pattern_match
  pattern_hits: [{
    pattern_id, fault_type, confidence, symptom_signature,
    verification_steps: list[str], solutions: list[str],
    source: "manual|learned", version: int, provenance_exp_ids: list[str]
  }]
```

### 5.4 DataAnalysisTool

```
输入 Schema：
  analysis_type: "correlation" | "changepoint" | "group_compare" | "lag_analysis"

  # correlation / lag_analysis：
  metrics_a: list[str], metrics_b: list[str]
  nodes: list[str], time_window: {start, end}
  max_lag_seconds: int (仅 lag_analysis)

  # group_compare：
  anomalous_nodes: list[str], normal_nodes: list[str]
  metrics: list[str], time_window: {start, end}

  # changepoint：
  metric: str, node: str, time_window: {start, end}

输出 Schema（结构化统计量 + 文字摘要，禁止原始序列）：
  analysis_id: str, analysis_type: str
  findings: [{metric_or_pair, statistic_name, value, interpretation}]
  summary: str  # 1-2句话的自然语言摘要，LLM 可直接使用
```

## Part VI：Knowledge Base（完整规格）

### 6.1 Metric KB

存储格式：YAML（人工可读/可编辑）+ 向量索引（ChromaDB，用于 pattern_match 模式的语义检索）

每个条目：
```yaml
- name: "node_cpu_seconds_total"
  subsystem: "cpu"
  description: "CPU 在各模式下累计运行时间，rate() 后得到使用率"
  unit: "seconds (需 rate() 换算为 %)"
  normal_range: {min: 0, max: 0.95, unit: "rate_fraction"}
  strength_thresholds:
    weak:   {condition: ">0.80", duration_sec: 30}
    medium: {condition: ">0.90", duration_sec: 60}
    strong: {condition: ">0.95", duration_sec: 300}
  related_faults: ["cpu_fullload", "cpu_partial_load"]
  common_misconceptions: "单节点高不代表故障，需结合 job 分配情况判断"
```

### 6.2 Fault Pattern Library

存储格式：JSON Lines（每行一条规则），方便 append 与版本追踪。

每条规则：
```json
{
  "pattern_id": "fpl_001",
  "fault_type": "cpu_fullload",
  "version": 1,
  "source": "manual",
  "provenance_exp_ids": ["exp_001_cpu_fullload"],
  "confidence": 0.9,
  "symptom_signature": {
    "leading_subsystem": "cpu",
    "required_metrics": ["node_cpu_seconds_total"],
    "optional_metrics": ["node_load1", "node_context_switches_total"],
    "temporal_pattern": "cpu_usage rises first, memory unaffected"
  },
  "verification_steps": [
    "confirm cpu_usage p95 > 95% for > 5 min",
    "confirm no co-occurring memory pressure (memory_usage < 80%)",
    "confirm load1 > num_cores * 0.9"
  ],
  "solutions": [
    "identify and kill runaway process",
    "check job resource limits in Slurm config"
  ]
}
```

规则写入规则（REFLECT 节点必须遵守）：
1. 系统在写入前计算 `symptom_signature` 的哈希，若与已有规则 cosine_similarity > 0.85，则为 update（版本号+1），否则为新增。
2. LLM 只生成 `symptom_signature` + `verification_steps` + `solutions`；`confidence` 由系统根据"该规则在历史评估中的准确率"计算，初始值 0.5。
3. 写入的规则进入"候选状态"，需要被 2 次以上成功诊断确认后才升为正式规则。

### 6.3 Baseline Profile

存储：每个实验条件（workload 类型 × 节点数）一个 JSON 文件。

```json
{
  "profile_id": "bp_npb_bt_4nodes",
  "workload": "NPB_BT",
  "node_count": 4,
  "metrics": {
    "node_cpu_seconds_total": {
      "mean": 0.65, "std": 0.08, "p5": 0.5, "p95": 0.78,
      "window_size_sec": 60
    }
  }
}
```

Triage 的 Z-score 计算使用对应 workload 的 Baseline Profile，若无匹配则使用 global profile（所有实验的聚合统计）。

## Part VII：FINALIZE & REFLECT

### 7.1 DiagnosisReport Schema

```json
{
  "run_id": "string",
  "anomaly_summary": "string",
```

### 7.2 REFLECT 节点规格

触发时机：FINALIZE 完成后，可同步（小型 KB）或异步（大型 KB）。

输入：DiagnosisReport + `chaos.csv`（若为评估模式）

步骤：
1. **Ground Truth 比对**（评估模式）：将 `root_causes` 与 `chaos.csv` 中的实际注入故障对比，输出 `hit@1` / `hit@k` 标记，写入 trace。
2. **规则提炼**：LLM 分析本次诊断的"有效症状签名"与"有效验证路径"，生成候选规则条目（按 §6.2 Schema）。
3. **规则写入**：系统执行相似度去重后，写入 FPL（候选状态）。
4. **规则退化**：若某规则在最近 5 次相关诊断中准确率低于 0.4，自动降低其 `confidence` 并标记为 deprecated。

## Part VIII：失败处理

| 场景 | 处理策略 |
|------|---------|
| 工具超时 | 记录 `status=timeout`，该工具返回标记为不可用证据，禁止计入支持性证据 |
| 工具返回空数据 | 写入 `missing` 字段，对应 `required_verification` 标记为 `unverifiable` |
| LLM 输出不符合 Schema | 最多重试 2 次，仍失败则输出降级报告（仅包含 FocusContext + "无法完成推理"原因） |
| 预算耗尽但未 GATE 通过 | `diagnosis_type=inconclusive`，输出 Top-2 候选假设 + 缺失的关键证据说明 |
| FPL 命中但验证步骤失败 | 不允许仅靠规则命中下结论，必须至少通过 1 条 `required_verification` |

## Part IX：评估方案

### 9.1 数据集

- 单故障实验：来自 `dataset_builder/data/formaltest/`，ground truth 为 `chaos.csv`
- 复合故障实验：`compositetest/` 子集，要求 `diagnosis_type=composite_fault` 且 `root_causes` 覆盖全部注入故障

### 9.2 评估指标

| 类型 | 指标 |
|------|------|
| 准确性 | Root Cause Hit@1, Hit@3；Composite 故障全覆盖率 |
| 效率 | End-to-end 诊断时间；平均工具调用次数 |
| 成本 | 每次诊断 token 消耗（in/out 分开）；API 费用估计 |
| 鲁棒性 | 在缺失 20% 指标数据时的 Hit@1 下降幅度 |
| KB 增益 | 有无 FPL 命中的诊断，Hit@1 差异（RAG 效果） |

### 9.3 消融实验（论文必备）

| 消融维度 | 对应设计模块 | 消融方式 |
|----------|------------|---------|
| Ablation-A | 无分诊（Triage） | 跳过 Stage 1，直接用全量 metrics 的统计摘要作为输入 |
| Ablation-B | 无 Evidence Gating | 移除 GATE 节点，允许任意时机 FINALIZE |
| Ablation-C | 无 Knowledge Base | KBRetrievalTool 返回空结果 |
| Ablation-D | 无时序因果先验 | FocusContext 不包含 `causal_order`，打乱顺序 |
| Ablation-E | 无 Reflect（无 FPL 积累） | FPL 在诊断间不更新 |
| Baseline-1 | 单次 LLM 调用（当前系统）| `llm/src/main.py` 的现有实现 |
| Baseline-2 | 纯统计规则（无 LLM） | 仅 Triage + 硬编码规则匹配 |

消融 A/D 专门针对 v4 的新增贡献（时序因果先验），是区别于 v3 的核心对比。
