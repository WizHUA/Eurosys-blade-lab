# HPC-Diagnosis Agent Design (v5)

本文在 v4 基础上进行精简与迭代，面向 SC'26 论文写作优化设计表达，同时修正若干架构问题。

---

## Part I：v4 → v5 变更摘要

### 变更总览

| # | 变更内容 | 理由 |
|---|---------|------|
| 1 | **删除 Baseline Profile**，KB 简化为两层（Metric KB + FPL）；Triage Z-score 改为数据自适应 | 真实部署无预计算基线；减少对离线数据的依赖 |
| 2 | **FPL 初始为空**，不预填任何规则 | 证明系统可从零学习故障模式；消融对比更干净（空 vs 积累 vs 不用） |
| 3 | **Triage 6 步合并为 3 步** | 减少设计冗余，突出核心处理逻辑 |
| 4 | **明确 THINK → GATE 路由决策者** | v4 未定义"谁决定进入 GATE"，v5 给出清晰路由机制 |
| 5 | **Evidence Gating 精简**，不暴露具体阈值 | 论文写作友好；阈值作为可配置实现细节 |
| 6 | **Reflect 不依赖 Ground Truth**，GT 比对移入独立评估章节 | 真实环境无 chaos.csv；Reflect 应为无监督积累 |
| 7 | **消融实验精简为 4 ablation + 2 baseline** | 聚焦大块设计贡献，小粒度消融降为 supplementary |

### 设计原则（不变）

1. **两阶段清晰分离**（Triage + Agent），逻辑独立、消融边界明确
2. **ReAct 推理循环**：每次工具调用前有 Thought，每次观察后更新假设
3. **多假设并行跟踪**：显式维护 N 个候选假设，各自独立积累证据
4. **工具输出摘要化**：防止 token 爆炸
5. **规则库有 Schema**：FPL 有固定 JSON Schema，LLM 只能按格式写入

| 方向 | 内容 | 格式 |
|------|------|------|
| 输入 | `metrics.csv` — 时序指标矩阵（~150 列） | CSV，时间戳索引 |

### 2.3 与 STELLAR (SC'25) 的对比定位

| 维度 | STELLAR (SC'25) | 本系统 |
|------|-----------------|--------|
| 应用目标 | I/O 参数自动调优 | 故障根因诊断 |
| 推理控制 | LLM 自由生成 + 反思总结 | **确定性 Triage** + LLM Agent（双层控制） |
| 证据验证 | 无 Evidence Gating | **规则驱动的证据门控**，强制量化支撑 |
| 多故障支持 | 单目标优化 | **多假设并行**，支持复合故障 |
| 知识库 | RAG 检索文档片段 | **结构化 FPL**（含置信度、版本、来源追踪） |

### 2.4 核心贡献点

1. **Hybrid Control**：确定性 Triage（压缩 context，提取时序因果先验）+ LLM Agent（聚焦推理），两阶段逻辑独立
2. **Evidence Gating**：结论必须有量化证据支撑，防止 LLM 在证据不足时强行下结论
3. **Fault Pattern Library**：从空开始，通过 Reflect 自动积累的结构化故障知识库
4. **多假设并行 + 继发症状甄别**：显式维护 N 个假设，cascade 检测区分"复合故障"与"继发症状链"

## Part III：Stage 1 — Triage

Triage 是纯确定性 Python 代码，不调用 LLM，不消耗 token。职责：从 150+ 原始指标中筛选 Top-K 异常指标，输出带时序因果先验的 `FocusContext`。

### 3.1 处理流程（3 步）

#### Step 1：异常指标筛选（Anomaly Scoring）

对每个指标列执行：

- **数据自适应基线**：使用实验自身时序数据计算统计基线（前 N 个采样点的均值/标准差，或整体滑动窗口统计量），不依赖外部预计算的 Baseline Profile
- **Z-score 异常评分**：相对于数据自适应基线计算 Z-score
- **持久性过滤**：连续超阈值比例未达到持久性要求的指标剔除（去除毛刺）
- **变化点检测**（可选增强）：对候选异常指标做 PELT/BOCPD，标注 `change_point` 时刻用于精确定位 `t_onset`

输出：Top-K 异常指标，每个指标附带 `score`、`direction`（+/-）、`t_onset`（首次超阈时刻）

> **v5 变更**：Z-score 基线由 v4 的 Baseline Profile 查表改为数据自适应计算。这更符合真实部署场景——你不可能预先拥有每种 workload 在每种节点配置下的预计算基线。变化点检测融入本步骤，不再单独列为一步。

#### Step 2：时序因果排序（Temporal Ordering）

- 按 `t_onset` 升序排列异常指标，生成 `causal_order`
- 按 Metric KB 的 `subsystem` 字段分组，计算 `subsystem_scores`（各子系统加权异常总分）
- 确定 `leading_subsystem`（分诊初判的首发子系统）

#### Step 3：FocusContext 构建

聚合所有信息，生成输入给 Stage 2 的完整上下文：

- 将 Top-K 指标、causal_order、subsystem_scores 打包
- 与 `jobinfo.csv` 做时间窗口交集，关联受影响的作业和节点
- 计算 `triage_confidence`（基于异常指标质量的综合评分）

### 3.2 FocusContext Schema

```json
{
  "run_id": "string",
  "anomaly_window": {"start": "ISO8601", "end": "ISO8601"},
  "top_metrics": [
    {
      "metric": "string",
      "subsystem": "string",
      "direction": "+|-",
      "score": 0.0,
      "t_onset": "ISO8601",
      "onset_rank": 1,
      "change_point": "ISO8601|null"
    }
  ],
  "causal_order": ["metric_A", "metric_B", "metric_C"],
  "subsystem_scores": {"cpu": 0.0, "memory": 0.0, "network": 0.0, "disk": 0.0},
  "leading_subsystem": "string",
  "nodes": [{"node": "string", "severity": "high|medium|low"}],
  "jobs": [{"job_id": "string", "overlap_ratio": 0.0, "node_set": []}],
  "triage_confidence": 0.0
}
```

关键字段：
- `causal_order`：按 t_onset 排序的指标列表，是时序因果先验——HYPOTHESIZE 依据此顺序推断根因 vs 继发症状
- `leading_subsystem`：分诊初判的首发子系统，引导假设生成的优先方向
- `triage_confidence`：分诊可信度，低时 HYPOTHESIZE 生成更多候选假设

## Part IV：Stage 2 — Diagnosis Agent

### 4.1 AgentState Schema

```python
class AgentState(TypedDict):
    run_id: str
    inputs: dict                    # {metrics_path, jobinfo_path}
    focus_context: FocusContext     # Stage 1 产物
    hypotheses: list[Hypothesis]   # 多假设并行
    evidence: list[Evidence]       # 工具返回的摘要化证据
    react_trace: list[ReActStep]   # Thought-Action-Observation 轨迹
    budget: dict                   # {tool_calls_used, tool_calls_limit, tokens_in, tokens_out}
    gate_result: dict              # {decision, reason, diagnosis_type, ...}
    report: DiagnosisReport | None
```

### 4.2 Hypothesis / Evidence Schema

```python
class Hypothesis(BaseModel):
    id: str                         # h1, h2, h3...
    root_cause: str                 # 自然语言描述
    fault_type: str                 # 对应 FPL fault_type
    subsystem: str                  # cpu | memory | network | disk
    prior_confidence: float         # FPL 命中规则给出，无命中默认 0.30
    current_confidence: float       # OBSERVE 实时更新
    required_verifications: list[VerificationItem]
    status: Literal["active", "confirmed", "refuted", "derived"]

class Evidence(BaseModel):
    id: str
    hypothesis_ids: list[str]
    type: Literal["supporting", "refuting", "neutral"]
    source_tool: str
    query_summary: str              # 用于独立性去重
    result_digest: str              # 1-2 句话摘要（进入 LLM context）
    raw_stats: dict                 # 统计量（用于 Gate 规则，不进入 context）
    strength: Literal["strong", "medium", "weak"]  # Metric KB 规则确定性打标
```

### 4.3 LangGraph 节点与路由

#### 节点一览

| 节点 | 类型 | 职责 |
|------|------|------|
| HYPOTHESIZE | LLM | 基于 FocusContext + FPL Top-K 命中，生成 2–4 个候选假设 |
| THINK | LLM (ReAct) | 分析当前证据，决定下一步行动或尝试下结论 |
| ACT | 确定性 | 执行工具调用，记录消耗 |
| OBSERVE | 确定性 | 摘要压缩工具输出，打 strength 标签，更新 verification 状态 |
| GATE | 纯 Python | 评估结论证据是否充分 |
| FINALIZE | LLM | 按 DiagnosisReport schema 生成最终报告 |

#### 路由决策机制（v5 核心改进）

v4 未明确定义"谁在 THINK 之后决定走向 GATE 还是继续循环"。v5 明确如下：

```
                    ┌────────────────────────────────────────┐
                    │         BUDGET CHECK (确定性)           │
                    │  budget 耗尽? → 强制路由到 GATE         │
                    │  (remaining_budget = 0 flag)            │
                    └────────────┬───────────────────────────┘
                                 │ budget ok
                                 ▼
                    ┌────────────────────────────┐
                    │       THINK (LLM)          │
                    │  输出 action.type:          │
                    │  ┌─────────┐ ┌──────────┐  │
                    │  │tool_call│ │ conclude  │  │
                    │  └────┬────┘ └────┬─────┘  │
                    └───────┼───────────┼────────┘
                            │           │
                   ┌────────▼──┐  ┌─────▼─────┐
                   │  ACT      │  │   GATE    │
                   │ (工具执行) │  │ (规则判断) │
                   └────┬──────┘  └─────┬─────┘
                        │               │
                   ┌────▼──────┐   ┌────▼──────────┐
                   │  OBSERVE  │   │ decision =    │
                   │ (摘要压缩) │   │ "pass"? → FINALIZE │
                   └────┬──────┘   │ "continue"? → THINK │
                        │          └───────────────┘
                        ▼
                   → BUDGET CHECK → THINK（下一轮）
```

**路由规则说明：**

1. **THINK（LLM）决定意图**：THINK 节点输出时必须选择 `action.type`：
   - `"tool_call"` → 路由到 ACT → OBSERVE → 回到 BUDGET CHECK → THINK
   - `"conclude"` → 路由到 GATE（尝试结论判断）

2. **GATE（规则）验证结论**：
   - `"pass"` → 证据充分，路由到 FINALIZE
   - `"continue"` → 证据不足，路由回 THINK 继续循环

3. **Budget 检查（确定性）**：每次进入 THINK 前检查 budget，若耗尽则无论 THINK 想做什么，强制路由到 GATE（带 `remaining_budget=0` 标记）

**权责分离**：THINK（LLM）回答"我是否准备好下结论了"；GATE（规则）验证"你给出的结论证据够不够"。两者分离，防止 LLM 在证据不足时自信地下结论。

#### THINK 节点输出格式

```json
{
  "thought": "e1 显示 CPU p95=98.7% 持续 8 分钟，h1 的 verification[0] 已满足...",
  "action": {
    "type": "tool_call | conclude",
    "tool": "MetricQueryTool",
    "args": {},
    "reasoning": "需要验证 h1 的 verification[1]：是否无共发内存异常"
  }
}
```

### 4.4 复合故障支持

`AgentState.hypotheses` 列表可同时维护多个互不排斥的假设。THINK 节点在每轮综合所有 active 假设的证据状态，决定优先验证哪一个。

`diagnosis_type` 由 GATE 根据 confirmed 假设数量确定：
- `single_fault`：恰好 1 个 confirmed，无 active
- `composite_fault`：≥2 个独立 confirmed，均通过 cascade 甄别
- `partial`：≥1 个 confirmed，仍有 active（budget 耗尽）
- `inconclusive`：所有假设均 refuted 或 budget 耗尽无 confirmed

### 4.5 工具契约

#### MetricQueryTool

```
输入：nodes, metrics, time_window, aggregation (mean|p95|max|duration_above_threshold)
输出：results [{node, metric, value, unit}], missing [{node, metric, reason}]
注：LLM 接收的永远是聚合后的摘要，原始时序序列不进入 context
```

#### KBRetrievalTool

```
输入 mode A — metric_lookup：按指标名查询 Metric KB 条目（语义、正常范围、子系统）
输入 mode B — pattern_match：按症状签名查询 FPL（子系统 + 异常指标 + causal_order 线索）
输出：metric_entries[] 或 pattern_hits[]（含 confidence, verification_steps）
```

#### DataAnalysisTool

```
输入：analysis_type (correlation | changepoint | group_compare | lag_analysis)
输出：findings [{statistic_name, value, interpretation}], summary (1-2 句话)
注：禁止返回原始序列，只返回结构化统计量 + 文字摘要
```

## Part V：Knowledge Base

### 5.1 两层体系（v5 简化）

v5 将 v4 的三层 KB 简化为两层：

| 层次 | 存储格式 | 内容 | 变更 |
|------|---------|------|------|
| **Metric KB** | YAML + ChromaDB 向量索引 | 22 个关键指标的语义描述、正常范围、strength_thresholds、下游因果效应 | 不变 |
| **Fault Pattern Library (FPL)** | JSONL | 故障模式规则（症状签名 + 验证步骤 + 置信度） | **初始为空**，通过 Reflect 积累 |

> **v5 变更**：删除 Baseline Profile 组件。Triage 的 Z-score 基线改为数据自适应计算（见 Part III Step 1）。FPL 初始为空，系统完全从零开始积累规则。

### 5.2 Metric KB 条目规格

每个条目包含以下字段（详见 `design/kb_schema.md`）：

```yaml
- name: "cpu_usage_percent"
  subsystem: "cpu"
  description: "面向诊断的语义描述"
  unit: "%"
  normal_range: {min, max}
  strength_thresholds:
    weak:   {condition, min_duration_sec}
    medium: {condition, min_duration_sec}
    strong: {condition, min_duration_sec}
  related_faults: ["cpu_fullload"]
  downstream_effects:
    - metric: "memory_usage_percent"
      lag_range_sec: [120, 600]
      condition: "if_sustained"
```

关键设计：
- `strength_thresholds` 用于 OBSERVE 节点确定性打 strength 标签，不由 LLM 主观判断
- `downstream_effects` 用于 GATE 的 cascade 甄别

### 5.3 FPL 条目规格

```json
{
  "pattern_id": "fpl_xxx",
  "fault_type": "cpu_fullload",
  "version": 1,
  "status": "candidate | confirmed | deprecated",
  "source": "learned",
  "provenance_exp_ids": [],
  "confidence": 0.5,
  "symptom_signature": {
    "leading_subsystem": "cpu",
    "required_metrics": [],
    "optional_metrics": [],
    "temporal_pattern": "描述时序特征",
    "causal_order_hint": []
  },
  "verification_steps": [],
  "solutions": []
}
```

规则生命周期：
1. Reflect 生成候选规则，初始 `status: "candidate"`，`confidence: 0.5`
2. 被 ≥2 次成功诊断确认后升为 `"confirmed"`
3. 最近若干次相关诊断准确率低于阈值 → `confidence` 降低，标记 `"deprecated"`

> **FPL 初始为空的设计理由**：(1) 使消融对比更干净——空 FPL vs 积累后的 FPL vs 不用 FPL 三组对比；(2) 论文贡献更强——证明系统可以自行从诊断经验中学习故障模式，而不依赖人工预填规则。

## Part VI：Evidence Gating

Evidence Gating 是防止"LLM 编故事"的核心机制。GATE 节点是纯 Python 规则，不消耗 LLM token。

### 6.1 三项核心原理

**原理 1：证据独立性要求**

结论必须有来自不同来源（不同工具、不同指标）的独立支持证据。单一信息源不足以确认假设，即使信号很强——这防止了 LLM 把同一个现象反复包装成多条"证据"。

**原理 2：反驳优先**

强反驳证据可以直接否定假设，无论支持证据有多少。这体现了科学推理中"证伪优先"的原则——一个假设只要有一条无法解释的强力反例，就应当被放弃。

**原理 3：复合故障甄别**

当多个假设通过单假设评估时，需区分独立故障和继发症状链。判断依据包括时序先后性（几乎同时出现 → 独立注入）、因果链可达性（Metric KB downstream_effects）和异常幅度（继发效应通常幅度较小）。

### 6.2 三阶段 Gate 决策

```
Stage A: 单假设评估 → 每个假设独立判定 confirmed / refuted / active
    ↓
Stage B: 继发症状甄别 → 对 ≥2 confirmed 假设做 cascade vs independent 判断
    ↓
Stage C: 全局决策 → 输出 pass / continue / degrade + diagnosis_type
```

**Stage A：单假设评估**

对每个 active 假设，基于其关联的支持/反驳证据和 verification 完成状态，判定是否应升为 confirmed 或降为 refuted。核心规则：

- **独立证据规则**：需要来自不同来源的多条支持证据，且至少有一条达到 strong 强度
- **全面核实规则**：当所有 verification steps 均已完成且有 strong 证据时，可确认假设
- **反驳规则**：强反驳证据可直接否定假设

> 具体阈值（独立证据数量、FPL 先验要求等）为可配置实现细节，见 `config/gate_config.yaml`。

**Stage B：继发症状甄别**

当 ≥2 个假设被 Stage A 确认时，两两检查是否存在因果从属关系：

- **时序同步性**：两者几乎同时出现 → 独立故障（同时注入的特征）
- **因果链可达性**：查 Metric KB downstream_effects，若主假设不可能导致次假设的子系统异常 → 独立故障
- **异常幅度判断**：次假设的异常幅度达到 strong 级别 → 继发效应不可能达此幅度，判定独立故障
- 以上条件均不满足 → 判定为继发症状（h_secondary 降级为 `derived`）

**Stage C：全局决策**

基于 Stage A/B 结果输出最终决策：

| 条件 | 决策 | diagnosis_type |
|------|------|---------------|
| 所有 active 假设已收敛，≥1 confirmed | `pass` | single/composite_fault |
| 仍有 active 假设，budget 充足 | `continue` | — |
| Budget 耗尽，≥1 confirmed | `degrade` | partial |
| Budget 耗尽，无 confirmed | `degrade` | inconclusive |
| 所有假设 refuted | `degrade` | inconclusive |

## Part VII：Reflect

每次 FINALIZE 完成后触发 Reflect。核心职责：将本次诊断经验提炼为 FPL 规则，实现无监督知识积累。

### 7.1 执行步骤（不依赖 Ground Truth）

1. **规则提炼**：LLM 分析本次诊断的有效症状签名（leading_subsystem + anomaly metrics + temporal_pattern）和有效验证路径，按 FPL Schema 生成候选规则条目
2. **去重写入**：计算新规则 `symptom_signature` 与已有规则的相似度，高相似度则更新（版本号+1），否则新增。新规则初始 `status: "candidate"`，`confidence: 0.5`
3. **规则升级 / 衰减**：
   - 候选规则被后续诊断成功匹配并确认 ≥2 次 → 升为 `"confirmed"`
   - 最近若干次相关诊断中表现不佳 → `confidence` 降低，标记 `"deprecated"`

### 7.2 防污染措施

- LLM 只生成 `symptom_signature`、`verification_steps`、`solutions` 三个字段
- `confidence` 由系统根据规则在历史诊断中的表现自动计算，不允许 LLM 直接设置
- 写入前经过 Pydantic Schema 校验，不合法的规则被拒绝

> **v5 变更**：v4 的 Reflect 第 1 步是 Ground Truth 比对（用 chaos.csv），这在真实部署中不可用。v5 将 GT 比对完全移出 Reflect 流程，放入独立的评估框架（Part VIII）。Reflect 本身是无监督的——仅基于诊断过程中收集到的证据质量和一致性来提炼规则，不需要知道"正确答案"。

## Part VIII：评估方案

### 8.1 Ground Truth 比对（独立于 Reflect）

在实验评估模式下，使用 `chaos.csv`（记录了实际注入的故障类型、时间、节点）作为 Ground Truth：
- 将 `DiagnosisReport.root_causes` 与 `chaos.csv` 比对
- 计算 Hit@1、Hit@K、Composite Coverage 等指标
- 评估结果写入 ExecutionTrace，不影响 FPL 写入

### 8.2 数据集

| 数据集 | 规模 | 用途 |
|--------|------|------|
| formaltest | 29 实验（7 单故障 + 22 复合故障） | 主评测 |
| compositetest_backup | 8 实验（复合故障为主） | 补充评测 |

### 8.3 评估指标

| 类别 | 指标 | 说明 |
|------|------|------|
| 准确性 | Root Cause Hit@1 | Top-1 预测与 GT 匹配率 |
| 准确性 | Root Cause Hit@3 | GT 在 Top-3 候选中的覆盖率 |
| 准确性 | Composite Coverage | 复合故障中所有注入故障均被诊断出的比例 |
| 效率 | Avg Tool Calls | 每次诊断的平均工具调用次数 |
| 效率 | End-to-end Latency | 从输入到输出的总耗时（秒） |
| 成本 | Token Usage (in/out) | 每次诊断的 token 消耗 |
| 鲁棒性 | Hit@1 @ 20% missing | 随机删除 20% 指标列后的 Hit@1 下降幅度 |
| KB 增益 | ΔHit@1 (with/without FPL) | Abl-C vs Full 的 Hit@1 差异 |

## Part XI：失败处理

| 场景 | 处理策略 |
|------|---------|
| 工具超时 | 标记为不可用证据，禁止计入支持性证据 |
| 工具返回空数据 | 对应 verification 标记为 `unverifiable` |
| LLM 输出不符合 Schema | 最多重试 2 次，仍失败则输出降级报告 |
| Budget 耗尽未 GATE 通过 | `diagnosis_type=inconclusive`，输出 Top-2 候选假设 |
| FPL 命中但验证失败 | 不允许仅靠规则命中下结论 |
